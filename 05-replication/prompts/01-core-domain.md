# M2 核心域:门店 / 员工 / 权限

> 本页是复刻路线第二步(M2)的操作指令:在 M1 骨架上建立门店、总部用户、门店员工三张核心表及其生命周期,并提前拆掉「在职语义」「双身份鉴权」「统计口径」这三颗我们亲手踩响过的雷。适合准备动手的工程师,以及被整块喂进指令的 AI 编程助手。

**读完你会知道:**

- 为什么总部用户和门店员工必须分表,以及分表之后最容易埋下的那颗雷(在职字段语义方向)
- 门店生命周期怎么用状态机 + 口径文档管住,为什么闭店是状态不是删行
- 多角色选逗号串还是关联表,各自的取舍是什么
- 为什么存活率/闭店率的口径文档要在写任何统计代码之前先落纸
- 一段可以整块复制给 AI 的指令,和一份可逐条勾选的验收清单

## 目标

在 M1 骨架之上,建立整个系统的「人与店」地基:

1. **门店表**:基础信息 + 生命周期状态机(筹备/营业/闭店),状态语义写入口径文档;
2. **总部用户表 + 门店员工表**:分表建模,两边「在职」字段语义方向强制一致;
3. **多角色与权限**:选定建模方式并把角色判断封装成唯一函数;
4. **双身份鉴权**:店端鉴权独立实现,与总部鉴权并存,各出一个 demo 接口;
5. **统计口径预埋**:存活率/闭店率的队列法口径文档先写好(公式 + 排除项),代码后行。

后续几乎每个模块——订货、库存、营业额、巡检、看板——都会 `shop_id → 门店表`、`user_id → 用户表` 地挂上来。这一步偷的懒,后面每个模块都要还,而且是带利息还。

## 前置依赖

- 已完成 [M1 骨架](00-bootstrap.md):框架跑通、统一响应约定、基础鉴权工具、定时任务框架就绪。
- 建议先读 [核心域:门店 / 员工 / 角色权限](../../02-modules/core-domain.md) 了解设计背景,再回来执行本页。

## 喂给 AI 的指令

下面这段整块复制给你的 AI 编程助手。它假设你的项目已有统一响应封装和口径文档的约定(M1 建立);如果口径文档还不存在,指令里会要求 AI 创建。

```markdown
# 任务:M2 核心域 —— 门店 / 总部用户 / 门店员工

你在一个已完成 M1 骨架的 Django 单体项目上工作(统一响应封装、基础鉴权工具、
定时任务框架已就绪)。本轮建立三张核心表及其生命周期,并预埋统计口径。
完成后同步更新项目的 CLAUDE.md 与口径文档(docs/CALIBER.md,M1 已建,
路径与文件名含大小写以 M1 定死的为准;若缺失,就按这个名字补建这一份,
绝不另起第二个口径文件——有两份的口径比没有还糟)。

## 一、门店表(Shop)

- 基础信息:门店名称、省/市/区、详细地址、经纬度(可空,预留给地图功能)、
  签约日期、开业日期、闭店日期、经营模式(直营/加盟,按需)。
- 生命周期状态机:至少三态——筹备中 → 营业中 → 已闭店。
  - 状态用整数枚举存储,每个枚举值的业务语义写进口径文档「门店状态」一节;
  - 只允许合法流转(筹备→营业、营业→闭店),非法流转(如已闭店→筹备中)
    在服务层函数里拒绝并返回明确错误,不要散落在视图里各写各的;
  - 将来若新增状态(停业整改、转让中等),必须同步更新口径文档,
    并写明该状态算不算「存活」——这直接影响后面的统计。
- 闭店是状态流转,不是删行。门店记录永不物理删除:历史订单、统计报表、
  员工记录都会引用它。

## 二、总部用户(HqUser)与门店员工(ShopStaff)分表

- 分成两张表:总部用户登录管理后台,门店员工隶属于某家门店(外键→Shop)、
  登录门店端。不要合成一张表——两类身份的登录方式、权限模型、生命周期
  完全不同,合表的结果是每个查询都拖着一串 if。
- 在职字段(本条是红线):两张表都要有在职/离职字段,字段名与语义方向必须
  完全一致——建议都叫 is_active,且 True = 在职。
  这是我们的反面教训:真实系统里这两张表历史上各自演化,一张用 1 表示在职、
  另一张用 0 表示在职,此后每一个写统计的人(包括 AI)都得先背这条口诀,
  背反了就是数据事故。把「在职字段语义」明确写进口径文档。
- 在职判断只允许存在一处实现:为两张表各封装唯一函数或 QuerySet 方法
  (如 manager 上的 .active()),业务代码一律调用封装,
  禁止在业务代码里直接比较字段值。将来语义或字段变化,只改这一处。
- 员工离职 = 状态流转,不删行(操作日志、历史单据、统计都还要引用他)。

## 三、多角色与权限

- 一个总部用户可能身兼多角色(比如既管运营又管选址)。两种建模,二选一并在
  口径文档写明选择与理由:
  - 逗号分隔字符串(roles = "ops,site"):查询简单、改动小,角色总量少
    (个位数)时够用;缺点是没有外键约束,按角色筛人用包含匹配时要小心
    前缀误中(查 "ops" 不能命中 "ops_admin"),这个处理必须收在封装函数里;
  - 关联表(UserRole):约束严谨、可加角色级属性,代价是多一次 join、
    改动面大。角色将来会很多或要挂属性时选它。
  我们实际用的是逗号串,至今够用——前提是下面这条封装铁律执行到位。
- 「某用户是否具有某角色」「筛出某角色的全部在职用户」各封装成唯一函数,
  全库只此一处实现,禁止散落的 split / contains。
- 权限从简:角色决定菜单与接口可见性;更细粒度的权限等具体模块需要时
  再加布尔开关字段(我们的做法:角色 + 按模块的权限位,没有上通用 RBAC
  全家桶)。M2 只打好「身份」与「角色」的地基。

## 四、店端鉴权独立实现

- 门店员工不在总部用户表里。店端接口绝不能复用总部登录校验——我们踩过:
  店端接口套了总部的登录装饰器,上线后门店端全员显示「未登录」。
- 为店端单独实现一套登录/会话校验(独立的装饰器或校验函数),
  与总部那套并存、互不掺和。
- 各写一个 demo 接口用于验收:
  - /api/hq/whoami:总部会话访问,返回当前总部用户信息;
  - /api/store/whoami:店端会话访问,返回当前员工与所属门店信息;
  - 用对方的会话访问必须被拒绝,返回统一的未授权错误。

## 五、统计口径预埋(先文档,后代码)

- 在口径文档新增「存活率/闭店率」一节,采用队列法(cohort):
  把同一时期开业的门店编成一个队列,看该队列到观察期末仍在营业的比例。
  - 存活率(N 个月)= 队列中开业满 N 个月且期末状态为「营业中」的门店数
    ÷ 队列门店总数;
  - 闭店率 = 1 − 存活率;
  - 排除项必须逐条写明:测试店、未真正开业即终止的筹备店、
    以及其他不参与统计的特殊状态;
  - 示例:2025 年 1 月开业队列 20 家,12 个月后仍营业 17 家,
    则 12 个月存活率 85%(示例数字,非真实数据)。
- 本轮只落口径文档,不实现统计接口。后续里程碑写看板时以此文档为唯一依据,
  代码与文档口径不一致时,以文档为准修代码。

## 红线(逐条遵守)

1. 两表「在职」语义方向必须一致;在职判断与角色判断只允许出现在封装函数里。
2. 门店与员工永不物理删除,生命周期全部走状态流转。
3. 店端与总部鉴权彼此独立;改任何鉴权代码,必须同时回归两种身份的
   「已登录可正常访问」与「未登录被拦截」——只验证匿名被拦是不够的。
4. 完成后更新 CLAUDE.md:三张表各自的职责边界、在职与角色的唯一判断函数名、
   两套鉴权入口的用法、口径文档的位置。
```

- [ ] 用总部会话访问店端接口、用店端会话访问总部接口,均被拒绝(权限隔离双向可验证)

- [ ] grep 全库:角色判断没有散落的 split / contains,全部经由唯一函数
- [ ] 口径文档包含「门店状态」与「存活率/闭店率」两节,后者含公式与逐条排除项

## 踩坑与红线

**在职语义两表相反**
- 症状:两边统计的人数对不上;新同事(和 AI)写查询时反复把在职当离职,报表隔三差五出错。
- 根因:两张表各自演化多年,一张用 1 表示在职、另一张用 0 表示在职,没有任何文档记录这件事,全靠口口相传。
- 铁律:语义方向建表时就统一(True = 在职),判断封装成唯一函数并写进口径文档;历史包袱改不动的,至少把口诀写进 CLAUDE.md 让 AI 每次都能读到。

**店端接口复用总部登录校验**
- 症状:门店端功能上线,门店全员显示「未登录」,总部这边自测却一切正常。
- 根因:门店员工根本不在总部用户表里,套用总部登录装饰器等于对店端全员查无此人;开发者只用总部账号自测,恰好测不出来。
- 铁律:两套身份两套鉴权入口;改鉴权必须双身份回归——不止验证匿名被拦,还要验证两种已登录身份都仍然能通过。

**统计口径后补**
- 症状:闭店率忽高忽低,老板问一次数换一个答案。
- 根因:先写了统计代码才回头定口径,测试店、没开成的筹备店混在分母里,不同人写的查询各排各的。
- 铁律:口径文档先行——公式和排除项落纸之后才允许写统计代码;文档是唯一真相,代码与它不一致就改代码。
