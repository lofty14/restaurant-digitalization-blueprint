# M4 选址工具 + CRM 招商

> 这一页是复刻提示词:一次搭起「拓店线」(选址评估 + 定铺 + 带看 + 审批)和「招商线」(线索管道 + 公海 + 看板 + 官网进线)两条业务线。适合已经完成 M1/M2 骨架、准备扩展总部管理能力的读者,把整块指令喂给你的 AI 编程助手即可开工。

**读完你会知道:**

- 选址工具的最小完整形态:主表 + 一对一分页详情 + 带看记录 + 多人审批流 + 地图打点
- CRM 招商为什么核心不是「表」而是「管道 + 公海」两套规则
- 对外线索写入接口(官网表单用)和内部接口为什么必须物理分离,以及怎么防刷
- 坐标系这种「口径级决策」为什么要在第一天就写进文档,而不是等地图偏移了再查
- 验收时该造多少假数据、按什么路径把全流程走一遍

## 目标

这个里程碑做两条互相独立、但都挂在「总部拓展」上的业务线:

1. **拓店线(选址工具)**:开发人员在外面跑商圈,把候选点位录入系统 → 分页填写定铺评估详情 → 登记带看记录 → 提交走多审批人审批流 → 全部点位在地图上打点总览 → 每周自动生成周报。解决的是「候选点位散落在个人微信和 Excel 里,老板看不到全貌」的问题。
2. **招商线(CRM)**:加盟意向线索从各渠道进来(含官网表单)→ 按可配置的阶段管道推进 → 每次跟进留时间线 → 超时未跟进的线索自动掉进公海、其他人可认领 → 漏斗与来源看板看转化。解决的是「线索压在个别销售手里,跟没跟进没人知道」的问题。

两条线共用一套鉴权与权限体系,但数据模型互不依赖,可以并行开发。

## 前置依赖

- **[M2 核心域](01-core-domain.md)必须先完成**:选址的「开发人」、CRM 的「跟进人/认领人」都是核心域的 User;审批权限、模块访问权限都挂在核心域的权限模型上。
- M1 骨架的响应约定(`code` 字符串 `'0'` 成功)、`loads_data`/`to_resp` 工具函数、session 鉴权、Celery Beat 定时任务基础设施,本页指令直接引用,不再重复定义。
- 不依赖订货/库存/财务等其他模块,顺序上可以和其他 M4 模块并行。**别被文件序号 11 骗了**——序号只是这份文件写得晚,施工顺序它属于 M4,排在生产成本(M5)、内账(M5)、AI 助手(M6)之前,以[路线图的里程碑表](../README.md)为准。

```markdown
项目已有:函数式视图 + loads_data(req)/to_resp(resp) 约定、code 字符串 '0' 表成功、

## 一、选址工具(路由前缀 /api/site/)

### 建模
1. Site(选址主表):
   - 行政区三级:province / city / district(字符串即可,不必建区划表)
   - 商圈名 business_area、等级 level(S/A/B 三档,CharField + choices)
   - 开发人 developer(FK → User)
   - 状态字段分开建,不要合并成一个:提交状态 submit_status、审批状态 approve_status、
     匹配状态(是否已匹配到加盟商)match_status,各自独立流转
   - 房东联系信息(姓名/电话,注意这是个人信息,列表接口默认不返回电话全文)
   - 经纬度 longitude / latitude(FloatField,可空——先录文字信息后补点位是常态)
   - 创建/更新时间
2. SiteDetail(定铺详情,OneToOneField → Site):
   - 这是一张长评估表,拆成分页式表单:用 page_index 把字段分成若干页(例如 6 页:
     基础信息/人流商圈/竞品环境/租金条件/工程条件/照片附件),前端按页编辑、按页保存。
   - 保存接口必须支持只更新某一页的字段(部分更新),不能整表覆盖——两个人
     可能同时在填不同页。
   - 截图/照片类字段存对象存储的 key,不存二进制。
3. TakeLookRecord(带看记录):FK → Site,带看人、带看时间、意向人信息、结果备注。
   一个点位可以有多条带看记录。
4. SiteApproval(审批):提交人发起 → 指派一个或多个审批人(approver_ids)→
   每个审批人独立通过/驳回 → 全部通过则 Site.approve_status 置为已通过,
   任一驳回则整单驳回并记录驳回理由。审批动作要留操作日志(谁、何时、结论、意见)。

### 坐标系口径(红线,先定后写码)
全项目统一使用一种坐标系:[GCJ-02(国内高德/腾讯地图系)或 WGS-84,按你对接的
地图服务选定]。选定后:
- 在口径文档 docs/CALIBER.md(M1 定死的全项目唯一一份)里新增「坐标系」条目:
  写明用哪个坐标系、经纬度字段的含义、前端选点组件必须输出同一坐标系;
- 所有写入 longitude/latitude 的入口只接受这一种坐标系,不做静默转换;
- 绝不允许一部分数据是 GCJ-02、另一部分是 WGS-84 混在同一张表里——
  这种混用在地图上表现为「有的点准有的点偏几百米」,排查成本极高。

### 接口
- /api/site/save:新建/编辑主表;/api/site/detail_save:按 page_index 保存详情某一页
- /api/site/list:列表 + 筛选(行政区/等级/开发人/状态),支持可选分页
  (传 page 才分页,不传返回全量——地图页和管理页共用)
- /api/site/map_markers:地图打点专用接口。只返回精简字段:
  id、名称、等级、经纬度、审批状态,别的一概不返回。列表页那套长字段
  (房东信息、详情、审批历史)绝不出现在这个接口里——地图一次拉全量点位,
  字段越多越慢,而且房东电话这类信息不应该跟着地图接口批量外流。
- /api/site/submit_approval、/api/site/approve:审批流两步。审批接口须校验
  当前用户在 approver_ids 里,且只处理自己名下未处理的审批项。
- /api/take_look/save、/api/take_look/list:带看记录。
- 所有 /api/site/* 接口统一走一个权限校验函数 check_site_permission(request, admin=False):
  校验登录 + 选址模块权限;审批类接口传 admin=True 校验审批权限。
  不要每个视图各写各的 if,权限口子必须单点。

### 周报任务
Celery Beat 定时任务(如每周一上午):汇总上周新增点位数、带看次数、审批通过/驳回数、
按开发人分组的明细,生成一段文字或表格,推送到 [你的 IM 群机器人 webhook] 或
落库供页面查询。数字为 0 的周也要发(「上周无新增」也是信息)。

## 二、CRM 招商(路由前缀 /api/crm/)

### 建模
1. Lead(线索表):
   - 姓名、电话(唯一性去重键之一)、意向城市、来源 source(FK 或 choices:
     官网表单/电话进线/转介绍/展会/自拓……来源要可枚举,后面看板按它分组)
   - 阶段 stage:不要写死在代码里。建一张 StageConfig 表(名称、排序、是否终态、
     是否算「赢单」),管道阶段可配置——今天是「新线索→初步沟通→意向确认→
     实地考察→签约/流失」,下个月运营就会想加一段,写死了每次都要发版。
   - 当前跟进人 owner(FK → User,可空——空即在公海)
   - 最近跟进时间 last_follow_time(冗余字段,公海回收任务靠它扫,别每次 join 跟进表)
2. FollowUpRecord(跟进记录):FK → Lead,跟进人、时间、方式(电话/微信/面谈)、
   内容摘要、下次跟进计划时间。前端按时间倒序渲染成时间线。
   每写入一条跟进记录,同步刷新 Lead.last_follow_time。
3. 公海机制:
   - 回收:Celery Beat 每日任务扫描「有 owner 且 last_follow_time 超过 [N=7,示例数字]
     天」的非终态线索,置 owner 为空、写一条系统跟进记录「超时回收进公海」。
   - 认领:公海列表任何有 CRM 权限的人可见;认领接口把 owner 置为自己,
     并开始 [M=7,示例数字] 天保护期——保护期内即使没有新跟进记录,回收任务也不碰它
     (实现上认领时把 last_follow_time 刷成认领时刻即可,或单独记 claim_time)。
   - 认领要防并发:两个人同时点认领,用 update(...).filter(owner__isnull=True)
     这类条件更新保证只有一人成功,失败方返回「已被认领」。
4. 阶段变更、认领、回收都写操作日志(谁、何时、从什么到什么)。

### 接口
- /api/crm/lead/list:服务端分页 + 筛选(阶段/来源/跟进人/城市/时间段)。
  必须是真·服务端分页(LIMIT/OFFSET 或游标),绝不允许全量拉回内存再切片——
  验收会用上万条数据压。给 last_follow_time、owner、stage、电话加索引。
- /api/crm/lead/save、/api/crm/follow/save、/api/crm/lead/claim(认领)、
  /api/crm/lead/change_stage
- /api/crm/dashboard/funnel:漏斗看板——按 StageConfig 排序输出各阶段线索数与
  阶段间转化率;/api/crm/dashboard/source:来源看板——按来源分组的线索数与赢单数。
  两个看板都支持时间段筛选。

### 对外线索写入接口(红线最多的一个)
官网「留资表单」要把线索写进来,单独做一个接口(如 /api/crm/public/lead_submit),
与内部接口物理分离,规则:
1. 只写不读:这个接口无鉴权(官网访客没有 session),因此它绝不返回任何已有数据——
   不返回「该手机号已存在」这种可用于撞库的信息,成功失败一律返回同一句「已收到」。
   查重去重在服务端静默做(同手机号短期重复提交合并为一条,追加备注)。
2. 防刷:按 IP + 手机号做频率限制(如同一 IP 每分钟 [3,示例数字] 次)、
   校验字段长度与手机号格式、加一个前端隐藏的蜜罐字段(有值即静默丢弃)。
   被限流也返回「已收到」,不给攻击者反馈信号。
3. 该接口的 URL 不复用 /api/crm/ 内部前缀的权限中间件豁免逻辑,单独注册、
   单独 code review——历史上「为一个公开接口给整个前缀开豁免」是常见事故源。
4. 新线索来源自动标记为「官网表单」,进公海(无 owner),由认领机制分配。

## 三、通用要求
- 电话号码在列表类接口默认脱敏(中间四位打星),详情接口按权限返回全文。
- 完成后更新项目 CLAUDE.md:新增选址与 CRM 两个模块的路由前缀、模型清单、
  权限校验入口函数名;更新口径文档:坐标系条目、公海回收/保护期规则条目
  (N 天回收、M 天保护期,写明当前值)。
- 为公海回收任务和对外写入接口写测试:回收边界(第 N 天不回收、第 N+1 天回收)、
  保护期内不回收、并发认领只成功一个、蜜罐字段命中静默丢弃。
```

## 验收清单

- [ ] 选址审批流全链可走通:录入点位 → 填详情(至少两页分别保存,互不覆盖)→ 提交 → 指派 2 名审批人 → 一人通过一人驳回则整单驳回;重新提交后两人都通过则状态置已通过,操作日志完整
- [ ] 定铺详情按页保存:A 页保存不影响 B 页已填内容
- [ ] 带看记录可对同一点位登记多条,按时间倒序展示
- [ ] `/api/site/map_markers` 返回体只含 id/名称/等级/经纬度/审批状态精简字段,抓包确认没有房东电话等长字段
- [ ] 坐标系口径已写入口径文档,前端选点写入的坐标在地图上打点位置准确(抽查 3 个真实地址无偏移)
- [ ] 周报定时任务手动触发一次,内容包含上周新增/带看/审批统计,无数据周也正常输出
- [ ] CRM 阶段管道可配置:在 StageConfig 加一个新阶段,不改代码、不发版,列表和漏斗看板即生效
- [ ] 公海回收:造一条 last_follow_time 为 8 天前(示例数字,超过回收阈值)的线索,跑回收任务后 owner 清空并留系统记录;6 天前的不回收
- [ ] 认领后保护期生效:认领一条公海线索,不写任何跟进记录,保护期内再跑回收任务,该线索不被回收
- [ ] 并发认领:两个账号同时认领同一条线索,只有一人成功,另一人收到「已被认领」
- [ ] 线索列表万级数据分页流畅:脚本造 3 万条(示例数字)虚构线索,列表翻页与组合筛选响应在可接受范围(如 500ms 内,示例数字),确认 SQL 走了索引而非全表扫
- [ ] 对外写入接口安全:不带任何登录态可成功提交线索;用它尝试任何查询/枚举(重复手机号、改参数)都拿不到任何已有数据,重复提交与限流场景返回文案与正常成功完全一致
- [ ] 蜜罐字段填值提交,接口返回「已收到」但库里没有新增线索
- [ ] 列表接口电话已脱敏,详情接口按权限控制是否展示全文
- [ ] CLAUDE.md 与口径文档已更新:坐标系、公海规则、两模块路由与权限入口都能查到

## 踩坑与红线

- **地图上的点忽准忽偏**
  症状:大部分点位打点准确,少数偏出几百米,且偏移方向不固定。
  根因:两种坐标系(GCJ-02 / WGS-84)的数据混进了同一张表,通常是某个录入入口没走统一选点组件。
  铁律:坐标系全项目只认一种,写进口径文档;所有写入口只接受这一种,发现混入先治入口再洗数据。

- **公开写入接口变成数据泄露口**
  症状:官网表单接口为了「用户体验」返回了「该手机号已登记过」,被脚本拿去批量撞号。
  根因:无鉴权接口返回了与已有数据相关的差异化信息。
  铁律:对外写入接口只写不读,成功失败限流一律同一句回复;与内部接口分离注册,不共享任何权限豁免。

- **公海规则只存在于代码里**
  症状:销售抱怨「线索莫名其妙没了」,运营说不清回收是几天、保护期是几天。
  根因:回收/保护期阈值只写在任务代码里,没进口径文档,规则改过一次后口口相传的版本就错了。
  铁律:阈值当前值写进口径文档,每次调整同步更新;回收动作必须留系统跟进记录,让当事人看得到「为什么没了」。

- **分页详情整表覆盖**
  症状:两个同事分别填评估表的不同页,后保存的人把先保存的内容清空了。
  根因:保存接口拿整个对象做全量 update,而不是按 page_index 做字段级部分更新。
  铁律:分页表单的保存接口只更新本页字段;上线前必须做「两页交叉保存互不覆盖」的验收项。
