# M6 群聊客服机器人(聊天即操作)

> 这份指令让 AI 帮你搭一个能在群里「办实事」的客服机器人:店长发一句话就能录营业额、兑积分、领券,答疑走 FAQ + 知识库。设计蓝图见[业务型 AI:视觉巡检与聊天即操作](../../04-ai-engineering/business-ai.md)。

**读完你会知道:**

- 「聊天即操作」机器人的完整施工清单:绑定鉴权、意图路由、三个操作技能、红线双保险
- 为什么营业额、积分必须 100% 复用既有接口,新建的券模块也要把写入收口在自己的 service
- 验收时怎么用「重发消息」和「套话测试集」把幂等与红线真正测出来

## 目标

在企业 IM 群或微信群里上线一个机器人,支持:①自然语言答疑(FAQ 优先,未命中走大模型 + 知识库);②三个操作技能——录营业额、兑换积分、领券;③全程可审计。

## 喂给 AI 的指令

把下面整块复制给你的 AI 编程助手。消息通道以你实际使用的 IM 为准(企业 IM 的官方机器人 API,或微信生态的合规接入通道),通道差异只影响收发消息的适配层,不影响本指令的主体结构。

```markdown
请在现有项目上实现「群聊客服机器人的操作能力(Chat-Ops)」。你已经在 M6 第一步
(10-ai-assistant)建好了消息 webhook 接入(验签 + 平台事件 ID(event_id)去重)、
FAQ 知识库表、会话记录表;本步骤在其上增加"办事"能力——这正是 10 结尾预告的
「放开写能力」那一步:从「绝不写业务数据」升级为「白名单内的三个写技能」。
完成后同步更新 CLAUDE.md 与 docs/CALIBER.md(M1 已建,新增口径追加在这一份里,
不允许另建)。

## 第一步:成员绑定表(一切操作能力的鉴权地基)

- 新建绑定表:IM 平台用户标识 ↔ 内部员工 id + 门店 id,带绑定时间、状态(有效/解绑)。
- 绑定流程:未绑定用户向机器人发起任何操作指令时,拒绝执行并返回绑定引导
  (由管理员在后台录入绑定,或用户提供工号由管理员确认——选一种实现并写进文档)。
- 红线:操作类技能只对"绑定有效 + 消息来自登记过的自有群"的用户开放;
  群外私聊与未登记群一律只提供答疑,不提供操作。

## 第二步:意图路由

- 收到消息后按优先级路由:精确指令格式(如"录营业额 3860")→ FAQ 匹配 →
  大模型意图识别(输出:意图类型 + 参数 + 置信度)。
- 大模型识别置信度低于阈值(示例:0.7,写成配置)时,不猜——回澄清问题
  ("你是想录今天的营业额 3860 元吗?回复'是'确认")。
- 所有"办事"意图在执行前必须有一次确认交互(用户回复确认词才执行),
  除非消息本身是精确指令格式。

## 第三步:三个操作技能(写入全部收口,机器人层禁止直写库)

1. 录营业额:解析金额与日期(缺省=今天),校验绑定门店 →
   调用 05 已建的营业额手动录入接口(手动来源那一路,与既有录入入口
   同一个函数;若你的系统要区分堂食/外卖渠道,先在营业额模块加渠道
   字段,机器人不新造口径)。同店同日已有记录时,回提示并要求用户
   明确"覆盖"才更新。
2. 兑换积分:解析兑换目标 → 查询余额与可兑清单 → 用户确认 →
   调用积分唯一出口函数扣减(业务码=机器人兑换),流水自动落地。
3. 领券:这是三个技能里唯一需要新建业务模块的——此前的里程碑都没建过券。
   按业务模块规范新建:券模板表(名称/面额/有效期/每人限领数,面额为
   示例配置)+ 发放记录表(用户 + 模板唯一约束防重)+ 一个领取 service
   函数(券的全部写入收口在它里面)。机器人只是这个新模块的第一个调用方。
- ★铁律:营业额与积分必须复用既有业务函数/接口,禁止为机器人另写入库逻辑;
  新建的券模块把写入收口在自己的 service;机器人层(消息处理代码)不允许
  出现任何直接操作数据库的写语句——完成后 grep 自查。

## 第四步:红线双保险

- system prompt 写死数据红线:成本、采购价、毛利等敏感字段一律拒答,
  给出固定话术;只允许回答对外订货价与公开规则。
- 程序层拦截:机器人可调用的查询 service 单独收口,返回字段白名单化,
  敏感字段在 service 层就不返回——即使模型被套话,也无数据可漏。

## 第五步:防重与审计

- 消息层:按平台事件 ID(event_id)去重(10 已建);操作层:营业额同键覆盖需确认、
  积分兑换与领券按业务键防重(唯一约束 + 流水)。
- 会话记录表增加字段:识别出的意图、执行的动作、调用结果、耗时,
  供复盘与红线抽查。
- 交付降噪:一次请求最多回两条消息(确认 + 结果);执行过程不进群。

## 工程约定

- 全部接口遵循项目统一响应约定;操作动作写操作日志;
- 新增口径写入 docs/CALIBER.md:机器人录入的营业额与其他入口同源同口径、
  机器人兑换的积分业务码定义;
- 更新 CLAUDE.md:机器人模块位置、三个技能对应的接口、红线清单位置;
  并将 10 写入的「机器人只读业务数据,绝不写业务数据」铁律升级为:
  「机器人写操作仅限本步三个白名单技能,100% 走既有/收口的业务 service,
  办事必确认」——同时注明 10 的验收项「可写的表仅限会话记录表与 FAQ 表」
  自本步起由该新铁律取代。
```

## 验收清单

- [ ] 未绑定用户发送「录营业额 100」被拒绝并收到绑定引导;绑定后同一指令成功
- [ ] 同一条操作消息重发(模拟平台重复投递),业务只执行一次——积分不重复扣、券不重复发
- [ ] 录营业额后,用营业额模块既有查询接口核对,数据一致(同一接口同一口径);同店同日再录,收到覆盖确认提示
- [ ] grep 全库确认:机器人消息处理代码中不存在直接写库语句(营业额/积分走既有接口,券走新建 service)
- [ ] CLAUDE.md 中机器人写权限铁律已更新为白名单表述,不再保留「绝不写业务数据」的旧表述(避免文档与代码现状矛盾)
- [ ] 兑换积分的流水里能看到机器人专属业务码,金额与余额变动可对账
- [ ] 红线测试集(至少 10 条套取成本/采购价的问法,含拐弯抹角式)全部拒答,且查询 service 返回字段白名单里无敏感字段
- [ ] 低置信度意图收到澄清问题而不是被猜测执行;确认交互后才真正执行
- [ ] 会话记录表能查到:谁、何时、什么意图、执行结果;抽一条答疑和一条操作各核对一遍
- [ ] 机器人一次请求进群消息不超过两条(确认 + 结果)

## 踩坑与红线

- **给机器人单写一套入库逻辑**
  症状:机器人录的数据和其他入口口径对不上。
  根因:图快绕过既有接口。
  铁律:机器人是又一个客户端,写操作 100% 复用既有函数;验收清单里的 grep 项不是走过场。

- **靠 prompt 守红线**
  症状:模型被拐弯问法套出敏感数据。
  根因:只有 prompt 一层保险。
  铁律:service 层字段白名单是真正的保险,prompt 只是第一道礼貌拒绝。

- **没有确认交互直接执行**
  症状:模型把闲聊误判成操作意图,替店长「录」了一笔不存在的营业额。
  根因:意图识别的误判率被当成零。
  铁律:办事必确认(精确指令格式除外);低置信度必澄清。
