# M3 库存

> 本页是库存模块的复刻 prompt:四量库存模型 + 唯一变更出口 + 自动对账。给已经跑通订货商城(02)的团队,把「库存数字永远对得上」这件事一次做对。

**读完你会知道:**

- 为什么库存要拆成 count / prepare / on_way / usable 四个量,而不是一个数字
- 为什么库存变更必须收口到一个 service 函数,以及这个函数要长什么样
- 对账任务怎么设计:用流水重放去校验四量,而不是靠人肉盘查
- 盘点和订货联动分别怎么接进唯一出口
- 验收时怎么证明「全库没有第二处代码在改库存」

## 目标

在订货商城之上,建立一套**账实可核对**的库存体系:

1. **四量模型**——每个「商品 × 仓」维护四个数量字段,语义先写进口径文档再写代码;
2. **唯一出口**——所有库存变更收口到一个 service 函数,函数内部同步落流水表,流水就是库存的「账本」;
3. **自动对账**——定时任务用流水重放校验四量,任何绕过出口的脏写都会被抓出来告警;
4. **盘点与联动**——盘点盈亏、订货下单/发货/退货,全部作为唯一出口的调用方接入,不开任何后门。

这一步的价值不在功能多,而在纪律:库存是订货、生产、财务的共同地基,数字一旦飘了,上层全部失真。我们的经验是——**先立规矩(唯一出口 + 对账),再堆功能**,顺序反了后面追债追不完。

## 前置依赖

- **必须完成 [M3 订货商城](02-ordering-mall.md)**(和本页同属 M3):库存的主要变更来源就是订货链路(下单占用、发货扣减、退货回补),没有订单就没有联动对象。
- 需要 [M1 骨架](00-bootstrap.md)提供的定时任务机制(对账任务要挂上去)和统一响应约定。
- 需要 [M2 核心域](01-core-domain.md)的门店/仓与权限体系:盘点、库存调整都是有权限边界的操作。
- 建议先读结构篇 [库存:四量模型与自动对账](../../02-modules/inventory.md),理解为什么这样设计,再喂 prompt。

## 喂给 AI 的指令

把下面整块复制给你的 AI 编程助手。指令里 on_way 的记账视角默认与结构篇([库存:四量模型与自动对账](../../02-modules/inventory.md))一致——收货方不做库存,on_way 记在发货仓;若你的收货方也要建库存账,按指令第 0 步里的多仓视角整体替换,别两种混用。

````markdown
你现在在既有的订货系统之上实现库存模块。这个模块的核心不是功能数量,
而是一条纪律:库存的每一次变化都必须可追溯、可重放、可对账。请严格按
以下顺序实施。

## 第 0 步:先写口径文档,再写代码

在项目的口径文档(docs/CALIBER.md,若无则新建)中新增「库存四量」一节,
先把语义定死:

- count:在库量。仓库里账面上的实物数量。
- prepare:占用量。已被订单锁定、尚未实际出库的数量。
- on_way:在途量。已从上游发出、尚未确认入库的数量。
- usable:可用量。前台下单时校验的量,与 count/prepare 的换算等式
  (例如 usable = count - prepare)必须在口径文档中写成唯一公式,
  所有代码与报表都引用这一个定义,不允许各处自算。

每个业务动作影响哪些量、方向如何,用一张表写清。其中 on_way 的
记账归属必须先做一个显式决策——两种仓储拓扑二选一,写进口径文档:

- **收货方不做库存(单仓视角,本指令默认)**:on_way 记在发货仓。
  下单 → prepare 增;发货 → 本仓 count 减、prepare 减、on_way 增;
  收货确认 → 只清本仓 on_way,不给任何仓加 count(收货方没有库存账);
  退货确认入库 → 发货仓 count 回补。
- **收货方也做库存(多仓视角)**:on_way 记在目的仓。发货 → 发货仓
  count 减、prepare 减,目的仓 on_way 增;收货确认 → 目的仓 on_way
  减、count 增。

两种都对,但全项目只能选一种,所有代码按同一张动作表走,绝不允许
一处按单仓、另一处按多仓各写各的。本指令后文默认按第一种(单仓视角)
写;若你选第二种,把口径文档和第 6 步的方向一并对应改掉。写完口径
再动手建模。

## 第 1 步:数据模型

1. 库存表:一行 = 一个「商品 × 仓」,包含四个数量字段
   count / prepare / on_way / usable(数值型,支持小数),外加更新时间。
   商品 + 仓 建唯一索引。
2. 库存流水表(账本,只增不改):每条记录包含——
   商品、仓、量类型(count/prepare/on_way/usable 之一)、变更前值、
   变更后值、delta、原因(枚举:下单占用/发货扣减/收货入库/退货回补/
   盘点盈亏/残量归零/人工修正等)、关联单据类型与单号、操作人、时间。
3. 盘点单 + 盘点明细表:盘点单有状态机(进行中/已提交/已完成);
   明细一行 = 一个商品的账面数、实盘数、盈亏差。

## 第 2 步:唯一出口 service 函数

新建独立的 inventory_service 模块,对外只暴露一个变更函数,签名类似:

    change_inventory(goods_id, warehouse_id, quantity_type, delta,
                     reason, related_order_type=None, related_order_no=None,
                     operator=None)

函数内部必须做到:

- 原子更新:用数据库层面的原子自增/自减(如 Django 的 F 表达式)或行锁,
  禁止「先读出来、内存里加减、再写回去」的竞态写法;
- 同一事务内落一条流水:变更前值、变更后值、delta、原因、关联单据缺一不可;
- 若量类型的变更会影响 usable(按口径文档的公式),在同一事务内一并维护;
- 微小残量自动归零:变更后若某量的绝对值小于阈值(示例:0.001,阈值写成
  常量并写进口径文档),自动将其归零,并额外落一条 reason=残量归零 的
  流水——归零也是变更,也要留痕;
- delta 为 0 直接拒绝;库存不存在的「商品 × 仓」按业务约定自动建行或报错,
  二选一并写进口径文档。

## 第 3 步:红线(写进你的 CLAUDE.md,永久生效)

除 inventory_service 的唯一出口函数外,任何代码——包括视图、定时任务、
管理命令、数据修复脚本——一律不得直接 update 库存表的四个数量字段。
需要改库存,只能调用唯一出口。今后任何人(包括 AI)在这个项目里写到
「更新库存」时,第一反应必须是调 change_inventory,而不是写 UPDATE。

## 第 4 步:定时对账任务

挂一个每日定时任务(用 M1 骨架的定时机制):

- 对每个「商品 × 仓 × 量类型」,从流水表重放:期初值 + Σdelta,
  得到「应有值」;
- 与库存表当前值比对,差异超过残量阈值的,生成告警(写告警记录 +
  推送到你们的告警通道);
- 对账只告警、不自动修数:差异必须由人确认原因后,以 reason=人工修正
  走唯一出口修复,修复本身留流水。
- 全量商品逐仓跑,注意分批查询,不要一次性把全表拉进内存。

## 第 5 步:盘点

- 接口:创建盘点单(选仓)→ 系统对该仓**全量商品**生成明细(注意:
  是全量遍历,不是当前列表页的一页数据)→ 录入实盘数 → 提交时逐条
  计算盈亏 → 确认后,每条差异以 reason=盘点盈亏 调唯一出口调整 count。
- 盘点期间该仓是否锁单,按你们业务定,决定写进口径文档。

## 第 6 步:与订货联动

在订货链路的三个节点接入唯一出口(在订单模块的状态流转处调用,
不要复制逻辑):

- 下单成功 → prepare 增(占用);
- 发货 → count 减、prepare 减、on_way 增(占用转在途;on_way 记在
  发货仓还是目的仓,严格按第 0 步选定的那种拓扑,不许在这里另起一套);
- 收货确认 → on_way 核销(单仓视角只清 on_way;多仓视角同时目的仓
  count 增);退货确认入库 → 发货仓 count 回补。方向一律以口径文档
  的动作表为准。

每次调用都带上订单号作为关联单据,让流水能反查到单。

## 接口约定

沿用项目统一响应约定(code '0' 成功)。至少提供:
库存查询(按仓/按商品,返回四量)、流水查询(按商品+仓+时间段,
支持按关联单号反查)、盘点单的建单/明细/提交/确认、对账告警列表。
写权限按 M2 的权限体系收口。

2. 更新口径文档:四量语义与公式、动作-量变化对照表、残量阈值、
````

## 踩坑与红线

**「顺手 update 一下库存」是最大的敌人**
- 症状:某商品可用量时不时飘,查代码发现某个数据修复脚本直接 UPDATE 了库存字段,没留流水。
- 根因:出口没收严,红线只在口头上,新代码(尤其 AI 生成的)默认会走最短路径直改字段。
- 铁律:红线写进 CLAUDE.md 和口径文档,验收用 grep 说话;对账任务是最后一道网,兜住所有绕过出口的脏写。

**盘点只遍历了列表页的一页**
- 症状:盘点单提交后,只有前 20 个商品有盈亏记录,后面的商品全部漏盘。
- 根因:生成盘点明细时复用了带分页的列表接口,拿到的是一页切片而不是全量。
- 铁律:盘点明细必须走全量查询,验收必须用「超过一页的商品数」来测,一页以内的测试等于没测。

**对账任务自动修数,越修越乱**
- 症状:对账发现差异后自动把库存改成重放值,结果把一笔尚未入账的正常业务当成脏数据「修」没了。
- 根因:对账时点和业务时点之间有在途事务,重放值不一定是「对的值」,机器分不清脏写和时间差。
- 铁律:对账只告警不修数;修数必须人工确认原因后走唯一出口,并留「人工修正」流水。
