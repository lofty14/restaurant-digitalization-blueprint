# M4 营业额 + 看板

> 这一页是复刻里程碑 M4 的营业额模块 prompt:店端录入营业额、用录入率考核倒逼数据质量(不达标自动锁订货)、再在干净数据上盖看板。适合已经完成 M3 订货商城(含订货锁框架)的读者,把整页指令块直接喂给你的 AI 编程助手。

**读完你会知道:**

- 营业额模块为什么要按「录入 → 考核 → 看板」三层来建,而不是先建看板
- 「一店一天一条」和「有限修改窗口」这两条约束如何从建模层面杜绝脏数据
- 录入率达标锁如何作为一个锁条件挂进 M3 订货商城里已经建好的订货锁框架,而不是另起炉灶
- 看板接口为什么必须全部引用同一份「实收」口径定义,以及不这么做会付出什么代价
- 双来源(手动录入 + 外部导入)并存时,去重规则要提前预留在哪里

## 目标

本里程碑交付三样东西:

1. **营业额录入**:门店端每天录一条营业额,数据结构和修改规则都收紧,保证底层数据可信。
2. **录入率考核锁**:定时任务统计近 N 天录入率,不达标的门店自动锁订货——用业务杠杆逼数据完整,而不是靠运营天天催。
3. **看板**:日报、月报、门店排行、环比,全部建立在同一份口径之上。

先说一个顺序问题:很多团队上来先做看板,因为看板是老板最想看的。我们的教训是反过来——**看板只值它底下数据的钱**。录入没约束、考核没抓手,看板上的数字就是装饰品,老板看两周发现对不上账,整个系统的信用就塌了。所以这一页的指令块把大部分篇幅花在录入约束和考核锁上,看板反而是水到渠成的最后一步。

录入率考核锁的思路很简单:门店不录营业额,就不能订货。订货是门店的刚需,这个杠杆比任何催办消息都管用。我们上线这个锁之后,手动录入门店的录入率长期稳定在高位——不是因为店长变自觉了,是因为不录就订不了货。

## 喂给 AI 的指令

把下面整块复制给你的 AI 编程助手。方括号里的参数按你的实际情况替换。

```markdown
你在一个已有的 Django 单体项目上继续开发(已有:响应约定、session 鉴权、
定时任务框架、门店/员工/权限模型、订货商城及其订货锁框架)。
本次任务:营业额录入 + 录入率考核锁 + 营业额看板。

## 一、营业额录入(店端)

建模 TurnoverRecord(营业额记录):
- 字段:shop_id、biz_date(业务日期)、实收金额、来源 source
  (1=手动录入 / 2=外部导入,预留枚举空间)、录入人、创建/更新时间。
- 铁律①:一店一天一来源只能有一条记录。用数据库唯一约束
  (shop_id + biz_date + source)兜底,不要只靠应用层判断——
  并发提交时应用层判断挡不住。
- 铁律②:金额字段用 Decimal,不用 Float。

录入接口(店端,走店端鉴权):
- 提交:传 shop_id、biz_date、实收金额。同一天重复提交视为修改。
- 修改窗口:只允许修改「当日往前 [X] 天」内的记录([X] 建议 3,
  示例数字,可配置)。超窗修改一律拒绝,返回明确错误文案,
  引导走线下申请流程。为什么收这么紧:历史营业额一旦可以随意改,
  月报、环比、考核全部失去意义,而且你永远查不清哪个数字是后改的。
- 每次修改留痕:至少记录修改人和修改时间;有余力就建一张
  变更流水表,存修改前后的值。

## 二、录入率统计任务

建一个定时任务(挂在项目既定的定时系统上,别新开一套):
- 每日凌晨跑一次,对每个「需要手动录入」的门店,统计近 [N] 天
  ([N] 示例取 14,示例数字,必须做成可配置参数)的录入率:
  录入率 = 已录入天数 / 应录入天数。
- 「应录入天数」要排除:开业前的日期、明确标记停业的日期。
  新开门店不满 [N] 天的,按实际开业天数计算,不足最小考核天数
  ([M] 天,可配置)的直接跳过不考核——新店期误锁是最伤士气的 bug。
- 统计结果落表(shop_id + 统计日期 + 录入率 + 是否达标),
  不要只算不存:落表才能支持看板展示和事后追查。
- 铁律③:任务必须支持按指定日期重跑,且重跑幂等(同一天重跑
  覆盖旧结果而不是追加)。上线后你一定会遇到"某天任务没跑/
  跑错了"的情况,不能重跑的统计任务等于一次性任务。

## 三、达标锁(接入订货锁框架)

- 不要新写锁逻辑。在既有订货锁框架里注册一个新的锁条件:
  「录入率达标锁」——最近一次统计结果低于阈值(示例:70%,
  可配置)则触发锁。
- 锁的判定读统计结果表,不要在下单请求里实时算录入率
  (下单是高频路径,实时算又慢又容易和统计任务口径打架)。
- 豁免走订货锁框架的统一豁免入口,不要为这个锁单独开豁免接口。
  豁免要有操作人、原因、有效期,且只对「录入率达标锁」这一个
  条件生效,不影响其他锁条件。
- 门店被锁/解锁时,给出对店长可读的提示文案(告诉他为什么被锁、
  怎么解),不要只返回一个错误码。

## 四、看板接口

先在项目的《数据口径文档》里写清「实收」的唯一定义(哪个字段、
含不含哪些调整项),再动手写接口。以下所有接口的金额一律引用
这个定义,禁止任何接口自带一套 SUM 口径:

- 日视图:指定日期,全部门店的实收列表 + 合计。
- 月视图:指定月份,按天的实收序列 + 月合计。
- 门店排行:指定时间段,按实收排序的门店榜。
- 环比:本期 vs 上期(日环比/月环比),返回两期数值和变化率。
  分母为零时变化率返回 null,不要返回 Infinity 或报错。

实现要求:
- 把「按门店+日期区间取实收」封装成一个查询函数,四个接口
  全部调它。这是保证口径一致的工程手段——口径写在文档里,
  实现收敛到一个函数里,两边互为校验。
- 看板接口全部走管理端权限,门店端只能看自己店的数据。

## 五、双来源预留(本期只做结构,不做导入)

未来会有外部导入的营业额(平台抓取/POS 同步),现在就把合并
规则定下来,写进口径文档:
- 同店同日既有手动录入又有外部导入时,取哪条?定一个来源
  优先级(建议:外部导入 > 手动录入,因为机器数据可追溯),
  并把优先级做成配置而不是散落在代码里的 if。
- 所有看板查询函数在汇总前必须按「店+日」去重(按来源优先级
  取一条),绝不能两条都计入——这是双来源最容易翻车的地方。
- 本期不实现导入本身,但唯一约束、source 枚举、去重函数
  必须现在就位,否则将来导入上线时要回头改所有看板。
```

## 踩坑与红线

**看板数字对不上**
- 症状:日视图合计、月视图当日值、排行榜同一天的数,三个接口三个数字,业务同学拿着截图来问哪个是真的。
- 根因:每个接口各写一段聚合 SQL,有的含了某来源、有的没排重、有的日期边界差一天,口径在实现层悄悄分叉。
- 铁律:实收定义只存在于一处——口径文档 + 一个共用查询函数;新增看板接口只许调函数,不许自带 SUM。

**新店被误锁,店长第一周就对系统失去信任**
- 症状:新开门店刚上系统就被「录入率不达标」锁了订货,可它开业才几天,根本凑不满考核周期的数据。
- 根因:统计任务的「应录入天数」没有排除开业前日期,分母被放大,录入率被算低。
- 铁律:考核先定豁免边界(新店保护期、停业日排除),再定阈值;任何自动锁上线前,先跑一遍全量门店的模拟结果人工过目。

**统计任务跑坏一天,数据永久缺口**
- 症状:某天定时任务因故没跑,当天的录入率统计缺失,第二天锁的判定用了两天前的旧结果。
- 根因:任务只支持「跑今天」,没有日期参数,也没有幂等设计,补数只能手工改库。
- 铁律:所有统计类定时任务从第一天就支持按日期重跑且幂等;锁判定读最近一次有效统计,并对「统计过旧」有兜底策略。

**双来源上线日,营业额翻倍**
- 症状:外部导入功能上线当天,看板上部分门店营业额接近翻倍——手动录的和导入的各算了一遍。
- 根因:当初建模时唯一约束只覆盖了单来源,汇总函数没有按「店+日」去重。
- 铁律:去重规则在双来源真正到来之前就写进查询函数并配测试用例;唯一约束里必须含 source 字段,合并优先级写进口径文档。

## 验收清单

- [ ] 店端可为本店提交当日营业额;同店同日同来源重复提交表现为修改,数据库中只有一条记录
- [ ] 修改超出配置的窗口天数(示例 3 天)被拒绝,且返回可读的错误提示
- [ ] 录入率统计任务跑完后结果落表;修改「近 N 天」参数后重跑,结果随之变化
- [ ] 统计任务可传入历史日期重跑,重跑覆盖旧结果(幂等),不产生重复行
- [ ] 造一个录入率低于阈值的测试门店:统计任务跑完后,该店下单被订货锁拦截,提示文案说明原因
- [ ] 走统一豁免入口对该店豁免「录入率达标锁」后,该店可正常下单;豁免不影响其他锁条件的判定
- [ ] 新建一个开业不足最小考核天数的测试门店,不会被误锁
- [ ] 同一天的实收金额,在日视图合计、月视图当日值、门店排行中完全一致
- [ ] 环比接口在上期为零时返回 null,不报错
- [ ] 手工在库里造一条同店同日的外部导入记录后,看板金额不翻倍(按来源优先级取一条)
- [ ] 项目 CLAUDE.md 与口径文档已更新:实收定义、录入率公式、双来源合并优先级均可查到
