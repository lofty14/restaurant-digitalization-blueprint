# M4 积分 + 开收档 + 日报

> 这一页是复刻里程碑 M4 的第一块:把「积分体系 + 门店开收档打卡 + 工作日报」三个日常运营模块一次性交给 AI 建出来。适合已经跑完 M1/M2 骨架、准备给门店装上"每日节奏"的团队。

**读完你会知道:**

- 为什么积分必须只有一个入口函数,以及怎么用 grep 验证这条铁律没被破坏
- 开收档打卡的三要素:拍照上传、时间窗判定、短链直达(店端轻鉴权)
- 次日结算定时任务怎么做到幂等——重跑一百遍也不会重复发分
- 日报模块的两个关键约束:一人一天一篇的唯一性、已读回执的可查询性
- 一份可以整块复制给 AI 编程助手的完整指令,和一份可勾选的验收清单

## 目标

这三个模块看起来是三件事,实际上是一条链:

1. **积分体系**是激励的"账本"——所有奖惩最终都落到积分上,所以它必须像记账一样严谨:每一笔变动都有流水,且只有一个入口能改余额。
2. **开收档打卡**是门店每天的"心跳"——店长每天开档、收档各打一次卡(拍照为证),系统次日结算:按时打卡加分,缺卡超时扣分。这是积分体系的第一个大客户。
3. **工作日报**是管理层看门店的"窗口"——一人一天一篇,配已读回执和提交率看板,让"写了没人看"和"没写没人管"两个老毛病同时被治住。

三者一起做的原因很简单:打卡结算要调积分入口,日报和打卡共享同一套"每日节奏 + 次日盘点 + 看板红线"的模式。分开做会重复造轮子。

## 前置依赖

开工前确认以下几件事已就绪,否则先回去补课:

- **[M1 骨架](00-bootstrap.md)已完成**:统一响应格式(`code` 字符串 `'0'` 表示成功)、session 鉴权、Celery(或等价物)定时任务底座能跑。
- **[M2 核心域](01-core-domain.md)已完成**:`Shop`(门店)、`User`(员工)模型存在,能从请求里取到当前用户和其所属门店——打卡和日报都挂在"某店某人"上。
- **决定好打卡页的载体**:我们用的是小程序/H5 页面 + 短链直达,你可以换,但"店员掏出手机 10 秒内能打上卡"这个体验目标不能降。
- **想清楚积分的业务场景清单**:先列出你已知的加减分场景(开档、收档、日报、巡检整改……),每个场景给一个业务码。后面会不断新增,但码表机制要一开始就有。

## 喂给 AI 的指令

下面这块整体复制给你的 AI 编程助手。方括号里的内容按你的实际情况替换。

```markdown
你在一个已有 Django 单体项目上工作(M1 骨架 + M2 核心域已完成:统一响应约定、
session 鉴权、Shop/User 模型、Celery 定时任务底座)。本次任务:实现
「积分体系 + 门店开收档打卡 + 工作日报」三个模块。严格按下面的要求做,
做完同步更新项目的 CLAUDE.md 和口径文档。

## 一、积分体系:唯一出口原则(最高优先级)

1. 建两张表:
   - PointsAccount(积分账户):user_id、当前余额、更新时间。
   - PointsRecord(积分流水):谁(user_id)、何时(created_at)、
     因何(业务码 + 可读描述 + 关联业务单号)、变动(正负分值)、
     变动后余额。流水只增不改不删。
2. 写一个唯一出口函数,例如 points_service.change_points(
   user_id, delta, biz_code, biz_ref, remark),内部做三件事:
   校验参数 → 原子地更新余额 → 落一条流水。余额与流水必须在同一个
   数据库事务里,保证任何时刻"余额 = 流水累加"恒成立。
3. 建一张业务码表(常量或配置表):每个加减分场景一个稳定的业务码,
   例如 OPEN_SHOP_ON_TIME / CLOSE_SHOP_MISSED / DAILY_REPORT_SUBMIT
   之类。新场景必须先登记业务码再接入,禁止裸字符串。
4. 铁律:全项目除这一个函数外,任何代码不得直接写 PointsAccount 的
   余额字段或直接插 PointsRecord。视图、定时任务、后台脚本一律调用
   change_points。请在 CLAUDE.md 里写明这条,并注明验证方法:
   grep 全项目对余额字段的写操作,结果只应命中 points_service 一处。

## 二、开收档打卡

1. 建打卡表:shop_id、user_id、打卡类型(开档/收档)、业务日期、
   打卡时间、照片(拍照上传,存对象存储或本地介质,表里存引用)、
   判定结果(按时/超时/缺卡,结算时回填)。
2. 时间窗判定:每个打卡类型配置一个时间窗(如开档须在营业开始前后
   [X] 分钟内,示例配置,按你的营业时间定)。窗口内打卡记"按时",
   窗口外记"超时"。时间窗做成可配置,不要硬编码。
3. 打卡页体验:店员通过一条固定短链直达打卡页面,短链携带 shop_id。
   打卡页使用店端轻鉴权——不要求走完整的后台登录流程,用
   [店员身份的轻量校验方式,如小程序身份/店铺码] 确认"是这家店的人"
   即可。注意:店端员工可能不在后台 User 权限体系里,不要套用
   管理端的登录装饰器,否则店员会全线"未登录"。
4. 打卡接口要防重:同店同人同类型同业务日期重复提交,返回已打卡,
   不产生第二条记录。

## 三、次日结算定时任务(幂等是生命线)

1. 写一个定时任务,每天固定时刻(如次日午后,示例)结算前一天的打卡:
   - 按时打卡:调 change_points 加分(如 +[X] 分,示例数字);
   - 超时或缺卡:调 change_points 扣分(如 -[Y] 分,示例数字)。
2. 幂等设计(必须做,不是可选):
   - 判据用积分流水的业务唯一键:结算某人某天之前,先查
     PointsRecord 里"该 user_id + 该业务码 + 该业务日期"的流水
     是否已存在,存在即跳过。流水本身就是天然的幂等凭据,
     不要另建一套"发过没发过"的标记表;
   - 判据必须是业务唯一键,不是"任务今天跑过没有"这类
     易碎的运行时标记;
   - 任务重跑、Celery 重复投递、人工手动触发,都不得重复发分。
   验证方法:手动连续跑两遍结算任务,积分流水条数不变。
3. 定时任务注册到项目现有的调度系统里(M1 里定的那套),并在
   CLAUDE.md 记录任务名、调度时间、幂等机制。

## 四、工作日报

1. 建日报表:user_id、业务日期、正文内容、提交时间。
   在 (user_id, 业务日期) 上建数据库唯一索引——一人一天只能有一篇,
   第二次提交直接被数据库拒绝,接口层把这个错误翻译成友好提示。
   唯一性必须靠数据库约束兜底,不能只靠接口层判断(并发会穿透)。
2. 建已读回执表:日报 id、读者 user_id、已读时间。同一读者对同一篇
   日报只记一条。提供查询接口:给定一篇日报,列出谁读过、何时读的。
3. 管理看板接口:
   - 提交率:某天/某段时间内,应交人数 vs 实交人数;
   - 已读率:已提交的日报中,被目标读者(如上级)读过的比例。
   口径写进口径文档:分母是谁(全员?在职?某角色?)必须白纸黑字。
4. 命名注意:日报视图文件避免与框架或项目中已有的
   daily_report 之类名字冲突,起一个不会同名遮蔽的名字
   (先在项目里全局搜一遍候选名,确认无冲突再定)。

## 五、上传率看板与红线告警

1. 做一个打卡上传率看板接口:按门店 × 日期统计开收档打卡完成率。
2. 设一条红线阈值(如 [90]%,示例数字,按你的运营要求定):低于红线的门店在看板上
   高亮,并通过 [你的 IM 通知渠道] 告警到运营群。
3. 阈值可配置,告警要防刷屏(同一门店同一天只告警一次)。

## 通用约定与红线

- 响应格式、鉴权方式沿用 M1 约定,不要另起炉灶。
- 所有涉及"业务日期"的地方统一口径:打卡的业务日期是营业日,
  不是打卡动作发生的自然日(收档可能跨零点),把这条写进口径文档。
- 积分唯一出口铁律写入 CLAUDE.md;打卡时间窗、结算规则、看板分母
  口径写入口径文档;完成后把新增的接口清单也补进接口文档。
- 不要在代码里硬编码任何门店名、人名、真实分值——分值全部走配置。
```

## 踩坑与红线

这几条都是我们真实踩过的,写进指令还不够,验收时要专门盯:

**积分入口被绕过**
- 症状:某天发现余额和流水对不上,查了半天是一段后台脚本图省事直接 UPDATE 了余额字段。
- 根因:唯一出口只写在文档里,没有机制兜底,新代码(尤其 AI 生成的)很容易"顺手"直写。
- 铁律:每次合并涉及积分的代码,先 grep 余额字段的写操作;结果多于一处即打回。

**结算任务重复发分**
- 症状:调度系统抽风重复投递了一次任务,一批店员当天积分翻倍,回收积分比发积分难看得多。
- 根因:结算逻辑只管"算",发没发过没有可靠判据,天然不幂等。
- 铁律:发分前先查流水的业务唯一键(用户 + 业务码 + 业务日期),已存在即跳过;重跑必须是无害的。

**店端打卡页套了管理端登录**
- 症状:打卡页上线后所有店员集体"未登录",因为店员根本不在后台权限用户表里。
- 根因:图方便复用了管理端的登录装饰器,而店端和管理端是两套身份体系。
- 铁律:店端页面用店端轻鉴权;上线前必须用一个真实店员身份(而非管理员)完整走一遍打卡流程。

**日报唯一性只在接口层判断**
- 症状:偶发同人同天两篇日报,统计口径全乱。
- 根因:接口层"先查后插"在并发下必然穿透。
- 铁律:唯一性约束必须落在数据库唯一索引上,接口层只负责把冲突错误翻译成人话。

**业务日期与自然日混用**
- 症状:收档打卡在零点后提交,被算到了第二天,店长"无辜缺卡"被扣分。
- 根因:没有统一"营业日"口径,各处代码按自然日各算各的。
- 铁律:业务日期口径进口径文档,所有打卡/结算/看板代码只认营业日一个口径。

## 验收清单

- [ ] grep 全项目对积分余额字段的写操作,只命中 points_service 唯一出口一处
- [ ] 每笔积分变动都有流水,字段齐全:谁 / 何时 / 因何(业务码+单号)/ 变动 / 变动后余额
- [ ] 任取一个账户,流水累加值等于当前余额
- [ ] 业务码表存在,新增场景有登记入口,代码中无裸字符串业务码
- [ ] 开档/收档打卡可拍照上传,窗口内记按时、窗口外记超时,时间窗为可配置项
- [ ] 通过短链(带 shop_id)直达打卡页,店员用店端轻鉴权即可打卡,全程无需管理端登录流程
- [ ] 同店同人同类型同业务日期重复打卡不产生第二条记录
- [ ] 次日结算任务按规则加分/扣分;手动连跑两遍,积分流水条数不变(幂等验证)
- [ ] 结算任务已注册进 M1 的调度系统,任务名与调度时间记入 CLAUDE.md
- [ ] 同人同天提交第二篇日报被数据库唯一约束拒绝,接口返回友好提示
- [ ] 已读回执可查询:给定日报能列出读者与已读时间
- [ ] 看板接口返回提交率/已读率/打卡上传率,分母口径已写入口径文档
- [ ] 低于红线阈值的门店在看板高亮并触发一次(且仅一次)告警
- [ ] CLAUDE.md 已更新:积分唯一出口铁律、结算任务信息、店端/管理端双鉴权体系说明
- [ ] 口径文档已更新:营业日定义、时间窗规则、看板分母口径
