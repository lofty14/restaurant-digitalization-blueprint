# M3 订货商城

> 这一页是给 AI 编程助手的完整施工指令:从零建出「门店向总部订货」的全链路交易系统。适合已完成 M1/M2 的团队直接复制使用。

**读完你会知道:**

- 订货商城的交易主链条:商品 → 购物车 → 下单 → 订单状态机 → 退货/改单
- 为什么「价格快照」是整个模块最重要的一条设计决策,以及它如何写进指令
- 购物车并发、退货库存、改单退差价这三处最容易做错的地方,红线怎么提前埋进 prompt
- 一份可以逐条勾选的验收清单,验的全是我们真实踩过坑的地方

## 目标

这一步要建成的,是整套系统里**第一个真正的交易模块**:门店在小程序/管理端下订货单,总部审核、发货、结算,中间穿插退货、改单、锁单。它和 M2 的核心域不同——核心域错了顶多是数据难看,交易模块错了是**真金白银对不上账**。

所以这一页的指令块里,红线比功能多。功能写错可以改,口径写错(比如统计读了实时价而不是快照价)会在几个月后以「财务对不上」的形式爆发,排查成本是当初写对的几十倍。我们把这些教训全部前置进指令,让 AI 从第一行代码就走在对的路上。

建议的实施顺序(指令块里也是这个顺序):商品与类目 → 多规格 SKU → 购物车 → 下单与快照 → 订单状态机 → 退货单 → 改单退差价 → 订货锁框架 → 改价记录 / 归档 / 导出。前四步是地基,后面全部依赖它们。

## 喂给 AI 的指令

把下面整块复制给你的 AI 编程助手。假设它已经在 M1/M2 建好的 Django 项目里工作,且项目根目录有 CLAUDE.md。

````markdown
你在一个已完成骨架(M1)和核心域(M2)的 Django 单体项目里,为餐饮连锁建设「订货商城」模块:
门店向总部订货的完整交易链。请新建独立的 app(例如 `app/mall/`),模型、视图、路由都收在这个
边界内。全程遵守项目既有约定:函数式视图、`loads_data(req)` 取参、`to_resp(...)` 出参、
`code` 字符串 `'0'` 表示成功。

## 一、商品与类目

1. 类目模型:名称、排序、启用状态之外,必须带一个「配送方式」字段(例如 1=冷链、2=普通快递,
   整数枚举,后续可扩展)。这个字段不是展示用的——**下单时要按配送方式把购物车拆成多张订单**
   (冷链一张、快递一张),因为两种货走的物流、到货时效、签收流程完全不同。拆单逻辑写在下单
   服务层,不要塞在视图里。
2. 商品模型:名称、类目外键、单位、图片、上下架状态、排序。价格不直接放在商品主表上,
   放在 SKU 行上(见下一节)。

## 二、多规格 SKU:一规格一行

商品支持多规格(如 500g 装 / 2kg 装)。建模铁律:**一个规格一行记录,同一商品的多个规格
共用一个分组 id 字段**(例如 `summary_id`),而不是把规格塞进 JSON 字段或做成规格表×商品表
的多级嵌套。

- 好处:每个 SKU 有独立主键,购物车行、订单行、库存行、改价记录都直接外键到 SKU 行,
  没有任何一处需要解析 JSON;列表页按分组 id 聚合即可还原「一个商品多个规格」的展示。
- 商品列表接口返回时按分组 id 聚合;购物车与订单接口一律以 SKU 行主键为操作对象。

## 三、购物车:数量增减必须数据库层原子操作

购物车模型:门店 + SKU 唯一约束,数量字段。

红线:**数量的增减绝不允许「读出来、加一、写回去」**,那是并发丢更新的经典事故现场
(店长和店员同时在两台手机上加同一个商品,最后少了一份)。必须用数据库层的 delta 原子
更新,例如 Django 的 `F` 表达式:

```python
CartItem.objects.filter(shop_id=shop_id, sku_id=sku_id).update(count=F('count') + delta)
```

- 接口设计上,前端传的是 **delta(±1、±N)而不是目标值**,「设为 N」这种覆盖式写法
  只允许出现在明确的「修改数量」输入框场景,且也要走带条件的原子更新。
- 减到 0 及以下时删行。给这条路径写并发测试(多线程同时 +1,断言最终数量正确)。

## 四、下单:价格与商品信息快照进订单行

订单分主表(订单号、门店、下单人、状态、总额、配送方式)和订单行(SKU、数量、金额)。
注意外键指向:下单发生在门店端,下单人外键 → M2 的 `ShopStaff`(门店员工);后续审核、
发货、改单这些总部操作的操作人外键 → `HqUser`(总部用户)。M2 没有叫 `User` 的模型,
两类人各指各的表,不要混。

本模块最重要的一条红线:**下单那一刻,把价格和商品关键信息复制一份存进订单行**,字段统一
用 `snap_` 前缀:`snap_price`(下单时单价)、`snap_goods_name`、`snap_spec_name`、
`snap_unit` 等。此后这张订单的任何展示、统计、对账、导出,**只读 snap_ 字段,绝不回表
读商品实时价**。

为什么:商品价格会调整、名称会改、商品甚至会被归档。如果统计读实时价,昨天 10 元/串
(示例数字,非真实数据)下的单,今天改价成 12 元后,历史订单金额会凭空变化,财务月底
对不上账,而且这种错发现时往往已经过去几个月。

- 订单总额 = Σ(snap_price × 数量),在下单事务内算好落库,不做动态计算。
- 下单按类目的配送方式拆单(见第一节),拆出的每张订单各自独立走状态机。
- 你写完后自查:全项目 grep 订单相关的统计/导出/报表代码,确认没有任何一处 join 商品表
  取实时价参与金额计算。

## 五、订单状态机

用整数枚举字段 + 常量类定义状态:待审核 → 已审核/备货中 → 已发货 → 已完成,
外加 已取消。要求:

- 状态流转收敛到一个服务层函数(如 `transition_order(order, target_status, operator)`),
  内部校验「当前状态 → 目标状态」是否合法,非法流转直接抛错;视图层不允许直接改 status 字段。
- 每次流转记操作日志(谁、何时、从什么状态到什么状态)。
- 取消订单只允许在发货前;发货后要退,走退货单,不走取消。

## 六、退货单:独立状态机,仅退款不动库存

退货单是独立模型、独立状态机(申请 → 审核 → 完成/驳回),不复用订单状态字段。

- 退货单记录创建人、关联原订单与原订单行、退货数量、退款金额(以原订单行的 snap_price
  为基准计算,不读实时价)。
- 区分两种类型:「退货退款」(货要退回)与「仅退款」(货不退,只退钱)。
- 红线:**「仅退款」在审核确认通过之前,不允许触碰任何库存数量**;确认通过后也不产生
  入库(因为货没有回来)。「退货退款」的入库动作发生在确认收货那一步,同样不能在申请
  阶段就预扣或预增库存。库存联动先留出钩子函数,等库存模块就绪后再接。

## 七、改单退差价:以实收为锚点 + 互斥锁

订单发货前后,总部可能改单(缺货砍行、改数量)。改单产生的差价要退给门店。红线两条:

1. **以实收金额为锚点**:退差价 = 门店实际支付金额 − 改单后应付金额。绝不用「原订单
   金额」做锚点,因为一张订单可能被改多次、退过多次,只有「实收 − 当前应付」永远算得对。
   为此订单主表要维护「实收金额」与「累计已退金额」字段。
2. **并发加互斥锁**:两个运营同时改同一张单,或改单与退货同时发生,会把金额算花。对
   「同一张订单的金额变更操作」加互斥锁(数据库 `select_for_update` 或 Redis 锁均可,
   选一种并在注释里说明超时与释放策略),锁内完成「读实收 → 算差价 → 落库 → 记流水」全程。

每次改单/退差价写一条金额流水记录(原因、金额、操作人),让任何一张订单的钱都能被
逐笔还原。

## 八、订货锁框架:可插拔条件 + 统一豁免入口

业务上会有各种「不让某门店下单」的规则(例如:营业额录入率不达标、整改超期、费用逾期
——具体规则各家不同)。不要为每条规则写一个 if,要建框架:

- 定义「锁条件」接口:每个锁条件是一个独立的检查函数/类,输入门店,输出(是否锁定, 原因文案)。
  新增一种锁规则 = 新增一个实现并注册,不改下单主流程。
- 下单入口统一跑一遍所有已注册的锁条件,任何一条命中即拦截,返回原因文案给前端。
- **豁免走统一入口**:一个豁免模型(门店、锁类型、豁免时限、操作人、原因),所有锁条件
  在判定时都先查豁免表。绝不允许某条锁规则自己私设豁免逻辑——豁免口径一旦分散,
  运营同学就说不清「这家店到底为什么能/不能下单」。

## 九、改价记录 / 归档 / 导出

1. **商品改价记录**:每次 SKU 价格变更,写一条记录(SKU、旧价、新价、操作人、时间)。
   这既是审计线索,也是运营回答「什么时候调的价」的唯一依据。
2. **归档软隐藏 + 引用保护**:商品下架淘汰用 `is_archived` 布尔软隐藏,绝不物理删除
   ——历史订单行还外键着它。归档前做引用检查:存在未完结订单/退货单引用时禁止归档,
   给出明确提示。列表接口默认过滤已归档,但详情接口对历史单据仍可读。
3. **CSV 导出**:订单明细导出用 CSV 而不是 xlsx(生产容器未必装得起 Excel 库),
   输出带 UTF-8 BOM,否则中文在 Excel 里打开是乱码。导出金额一律取 snap_ 字段。

## 十、收尾要求

- 所有新接口按项目约定注册路由,前缀统一(如 `/api/mall/`)。
- 为购物车并发、状态机非法流转、仅退款不动库存、锁与豁免各写测试。
- **同步更新项目的 CLAUDE.md**:在业务模块索引里加上订货商城的落点(app 路径、路由前缀),
  并把三条口径红线写进去——「单据统计只读 snap_ 快照字段」「购物车增减必走数据库 delta
  原子操作」「仅退款确认前库存零变动」。
- 更新口径文档 docs/CALIBER.md(M1 定死的全项目唯一一份,绝不另建
  ORDERING_CALIBER.md 之类的第二份),新增「订货商城」一节:快照口径、实收锚点、
  锁与豁免的判定顺序,都要有一句话说明,让后来的人(和后来的 AI)不用读代码
  就知道红线在哪。
````

## 验收清单

每一条都对应我们真实付出过代价的坑,不要跳过:

- [ ] **改价不改历史**:下一张订单 → 把商品价格改掉 → 历史订单详情、列表金额、CSV 导出金额全部保持下单时的价格不变。
- [ ] **快照口径 grep 验证**:全项目 grep 订单统计/报表/导出相关代码,确认金额计算只引用 `snap_` 字段,没有任何一处 join 商品表读实时价。
- [ ] **购物车并发不丢更新**:有一个真正的并发测试(多线程/多请求同时对同一购物车行 +1),断言最终数量 = 初始值 + 请求次数;且代码里搜不到「读数量再写回」的模式。
- [ ] **仅退款库存零变动**:走完「申请仅退款 → 审核通过」全流程,前后对比相关 SKU 的所有库存数量字段,零变化(库存模块未接入时,验证钩子未被调用)。
- [ ] **状态机拒绝非法流转**:已发货订单调取消接口被拒;状态字段无法从视图层被直接改写。
- [ ] **拆单正确**:购物车同时包含两种配送方式的商品,下单后生成两张独立订单,各自金额正确。
- [ ] **锁与豁免闭环**:任意注册一个锁条件并使其命中 → 该门店下单被拦且返回原因文案;通过统一豁免入口加豁免 → 立即恢复可下单;豁免过期后重新被拦。
- [ ] **改单退差价对账**:对同一张订单连续改单两次,每次退差价金额 = 实收 − 当前应付,金额流水逐笔可还原;并发发起两次改单,互斥锁保证金额不花。
- [ ] **归档保护**:存在未完结订单引用的商品无法归档;归档后列表不可见,但历史订单详情仍能展示其快照信息。
- [ ] **CSV 中文不乱码**:导出文件用 Excel 直接打开,中文正常显示(UTF-8 BOM 生效)。
- [ ] **CLAUDE.md 与口径文档已更新**:模块落点、三条红线、快照/锚点/豁免口径均有记载。
