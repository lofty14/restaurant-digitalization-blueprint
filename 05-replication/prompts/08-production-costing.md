# M5 生产 + 成本

> 这一页是复刻路线图 M5 里程碑的第一个 prompt:让 AI 帮你搭起研发档案、成本卡、生产工单与批次追溯。适合已经完成库存模块(M3)、准备把中央厨房/工厂端管起来的团队。

**先说清楚:本篇只讲结构与模式,不含任何真实配方数据。** 下面指令块里出现的原料名、用量、价格全部是虚构示例。你复刻时喂给 AI 的也应该只是结构——真实配方数据永远只存在你自己的数据库里,并且锁在独立权限位后面。

## 读完你会知道

- 为什么研发档案要用一个**独立权限位**管起来,默认所有人不可见
- 成本卡的核心公式,以及那条让我们付出过代价的**量纲铁律**
- 生产工单、分步领料、批次效期、FEFO 出库之间怎么串成一条链
- 正反向追溯(原料批次 ⇄ 成品批次 ⇄ 出货门店)靠什么数据结构撑起来
- 一份可以整块复制、直接喂给 AI 编程助手的完整指令

## 目标

本里程碑交付四块能力,它们共享一套数据底座:

1. **研发档案**:配方(原料清单 + 用量)、工艺 SOP 的结构化存储,独立权限位保护。
2. **成本卡**:从配方自动算出单件成本,支持多级 BOM 逐级卷积;外采品走采购加权平均价。
3. **生产工单**:计划量、工序拆分、分步领料、产出入库带批次号。
4. **批次追溯**:效期管理、FEFO 出库、来料验收、正反向追溯查询。

这是全书里「机密浓度」最高的模块之一——配方就是餐饮企业的命根子。所以指令里对权限的要求会比其他模块严格得多:不是「管理员可见」,而是**默认所有人不可见,逐人授权**。

## 前置依赖

- **[M3 库存](03-inventory.md) 必须先完成。** 分步领料、产出入库、FEFO 出库全部要走库存模块的唯一变更出口;库存四量模型不稳,生产模块建上去就是沙上筑塔。
- 建议(非必须)先读结构篇 [生产与成本:研发档案 / 工单 / 批次追溯](../../02-modules/production-costing.md),理解每个设计决策背后的业务原因,再来喂指令。
- 成本口径会牵连订货商城的商品数据(外采品的采购价来自采购单),所以 [M3 订货商城](02-ordering-mall.md) 也应已就位。

## 一张图看懂追溯链

正反向追溯不是一个孤立功能,而是把生产全链路上每个环节都打上批次号之后的**自然结果**:

```mermaid
flowchart LR
    A[来料验收单] -->|生成原料批次| B[原料批次]
    B -->|工单领料记录| C[生产工单]
    C -->|产出入库| D[成品批次]
    D -->|订货出库记录| E[出货门店]
    E -.反向追溯.-> D
    D -.反向追溯.-> B
```

只要「验收→领料→产出→出库」每一步都落了带批次号的记录,追溯就是几条 JOIN 查询;哪一步偷懒没记批次,整条链就断在那里。

## 第零步:先写文档,再写代码

在写任何模型之前,先在口径文档 docs/CALIBER.md(M1 定死的全项目唯一一份,
绝不另建 COST_CALIBER.md 之类的第二份)新增「成本口径」一节,写清楚:

- **自产品成本 = 成本卡计算结果**(公式见下),绝不手填。
- **外采品成本 = 近 N 次采购的加权平均价**(N 做成可配置,按自家采购频率定,
  此处示例取 3~5 中一个值,按业务定),按采购数量加权,不是简单平均。
- **量纲铁律:任何原料单价参与成本计算前,必须经换算率(rate)转换到配方所用单位。**
  例如采购按「箱」计价、配方按「克」用料,则单价必须先除以每箱克数。
  这条要在文档里加粗写死,并注明「违反此条产生的成本数据全部作废」。
- 每种口径给一个虚构的手算示例,让后来的人能对着算一遍。

文档写完后同步更新项目 CLAUDE.md:在业务模块索引里加上生产模块的入口、
口径文档中「成本口径」一节的位置、以及研发档案的权限说明。之后每次改动接口
都要同步这两份文档。

## 一、研发档案(权限最严的模块)

### 权限
- 在 M1 建立的布尔权限表上新增独立权限位(如 rnd_permission 字段),**默认所有用户为关**。
- 研发档案下所有接口(列表/详情/编辑/成本卡)统一在入口处校验此权限位,
  无权限一律返回 HTTP 403,不返回任何字段级信息。
- 不要复用「管理员」之类的宽泛权限——配方是最高机密,必须逐人显式授权。

### 建模
- **RndArchive(研发档案主表)**:关联商品、状态(草稿/已发布)、更新人、更新时间。
- **RndIngredient(配方行)**:所属档案、原料(关联商品/原料主数据)、用量数值、
  用量单位(自由文本或单位字典均可,但必须存)、备注。
  一份配方 = 一组配方行,即「原料清单 + 用量」。
- **多级 BOM**:配方行的「原料」允许指向另一个自产品(它自己也有配方)。
  即 A 的配方里可以含半成品 B,B 又有自己的配方。建模时要防环
  (A 含 B、B 含 A 必须在保存时报错)。
- 工艺 SOP(工序步骤文本/图片)挂在档案下,结构自定,同受权限位保护。

### 成本卡计算
- 核心公式(单件成本):

  单件成本 = Σ(原料用量 × 原料单价) ÷ 出成率 ÷ 批量产出

  - 原料单价:自产原料递归取它自己的成本卡结果(**逐级卷积**),
    外采原料取加权平均采购价。
  - 出成率(yield rate):投入毛料到实际产出的比率,存在档案上。
  - 批量产出:一批投料产出多少件成品,存在档案上。
- **量纲铁律落地**:原料单价进入公式前必须调用统一的换算函数
  (如 convert_price(price, purchase_unit, recipe_unit, rate)),
  禁止在业务代码里散落手写除法。换算率缺失时必须抛异常,绝不静默按 1 处理。
- **为量纲换算写单元测试**:
  - 正确换算的用例(虚构数据手算对照);
  - **故意喂错单位/缺换算率的用例,断言必须报错**——这条测试不许省。
- 为多级 BOM 卷积写单元测试:构造一个两层 BOM 的虚构配方
  (成品含半成品、半成品含两种外采原料),手算出期望成本,断言计算结果一致。
- 外采品加权平均:写一个用例,虚构若干次采购(不同价格、不同数量),
  手算加权平均对照。

## 二、生产工单

- **ProductionOrder(工单)**:目标商品、计划量、状态机
  (草稿→已下达→生产中→已完工→已关闭)、所属工序列表。
- **工序(工单步骤)**:一张工单可拆多道工序,每道工序有自己的领料与产出。
- **分步领料**:每道工序独立创建领料单,领哪个原料、领多少、从哪个批次领;
  领料扣减库存**必须走库存模块的唯一变更出口**,禁止直接 UPDATE 库存表。
- **产出入库**:工单完工时按实际产出量入库,**必须生成批次号**
  (建议规则:商品编码 + 生产日期 + 序号,具体规则自定但要唯一、可读)。
  入库同样走库存唯一出口。

## 三、批次、效期与 FEFO

- **Batch(批次表)**:批次号、商品、生产日期、有效期至、当前剩余量、来源
  (生产工单产出 / 来料验收)。
- **效期**:有效期至 = 生产日期 + 商品保质期(保质期存在商品主数据上)。
- **FEFO 出库(First Expired, First Out)**:任何出库(门店订货发货、工单领料)
  分配批次时,按「有效期至」升序消耗,先到期的先出。
  把 FEFO 分配写成一个纯函数(输入:需求量 + 可用批次列表;输出:各批次扣减量),
  方便单测。
- 为 FEFO 写单元测试:虚构三个批次(到期日打乱顺序录入),
  断言出库分配顺序严格按到期日升序。

## 四、来料验收

- **IncomingInspection(来料验收单)**:采购单/供应商关联、原料、到货量、
  验收结论(合格/拒收/让步接收)、验收人、验收时间。
- 验收合格才生成原料批次并入库(走库存唯一出口);拒收不入库。
- 原料批次同样带效期,参与 FEFO。

## 五、正反向追溯查询

- **正向**:给一个原料批次号,查出它被哪些工单领用 → 产出了哪些成品批次
  → 这些成品批次发给了哪些门店。
- **反向**:给一个成品批次号,查出它用了哪些原料批次(含来料验收单与供应商),
  以及它发往了哪些门店。
- 实现要点:领料记录存「原料批次号 + 工单」,产出记录存「工单 + 成品批次号」,
  出库记录存「成品批次号 + 门店」——三张关联记录齐了,追溯就是 JOIN,
  不需要额外的「追溯表」。
- 提供两个查询接口:/api/production/trace/forward 与 /api/production/trace/backward,
  返回树状结构(层级:批次 → 工单 → 批次 → 门店)。

## 红线(违反任何一条即返工)

1. 研发档案相关接口,无权限用户必须 403,一个字段都不能漏。
2. 库存增减只能走库存模块唯一出口,生产模块里出现直接改库存量的代码即返工。
3. 单价换算必须走统一换算函数;缺换算率抛异常,不许默认按 1。
4. 成本卡结果是算出来的,不提供任何「手工覆盖成本」的后门接口。
5. 任何示例/测试/文档中的配方数据必须是虚构的,不得出现真实原料配比。
6. 所有出库路径(发货、领料)都必须走 FEFO 分配,不允许绕过批次直接扣总量。

- 跑全部单元测试并贴出结果。

## 为什么这几条要求写得这么死

**量纲铁律排在成本公式前面讲。** 我们在这条上真实地栽过:采购按大包装计价、配方按克用料,中间漏了一次换算,成本卡整体虚高了几个数量级——数字大到一眼假还好,最怕的是错得「看起来合理」。所以指令里不但要求统一换算函数,还要求**故意喂错单位的单元测试**,让错误在测试阶段就炸出来,而不是在老板看成本报表时。

**成本口径先落文档再实现。** 成本这个词在不同人嘴里指不同东西(采购价?卷积价?含损耗?)。文档先行,是逼着你在写代码前把「自产走成本卡、外采走加权平均采购价」这条口径定死,后面所有报表、所有对账都引用同一份文档,谁有疑问先看文档再吵。

**追溯不建「追溯表」。** 初学者容易想造一张大宽表记录全链路,但那是第二份真相,迟早和业务记录打架。正确做法是让领料、产出、出库三类业务记录本身带上批次号,追溯只是查询视角——这也是为什么前置依赖里反复强调库存唯一出口:所有变更走一个口,记录才齐。

## 踩坑与红线

- **症状**:成本卡算出的单件成本大到离谱或小到离谱。
  **根因**:采购单位与配方单位不一致,单价没经换算率转换直接进了公式。
  **铁律**:单价换算只走统一换算函数,缺换算率抛异常;喂错单位必报错的单测不许删。

- **症状**:多级 BOM 保存后成本计算卡死或栈溢出。
  **根因**:配方互相引用成环(A 含 B、B 含 A),卷积无限递归。
  **铁律**:保存配方行时做环检测,成环直接拒绝保存,不要指望计算时兜底。

- **症状**:追溯查询查到一半断链,某个成品批次查不到原料。
  **根因**:某条出入库路径绕过了批次记录(比如手工调库存、领料没选批次)。
  **铁律**:所有库存变更走唯一出口且强制带批次;发现绕行代码按红线返工。

- **症状**:研发档案接口在无权限账号下能拉到部分字段。
  **根因**:权限校验加在了「编辑」上,漏了「列表/详情/成本卡」等只读接口。
  **铁律**:权限校验统一放在该模块所有接口的入口处,新增接口默认继承,403 全覆盖。
