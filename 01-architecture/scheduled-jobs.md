# 定时任务:双系统并存的教训

> 这一页讲我们的定时任务体系怎么演化成「两套调度系统并存、几十个任务分散两处」的局面,以及如果重来一次会怎么做。适合后端负责人和准备用 AI 复刻的工程师读——这是少数「第一天做对成本几乎为零、后面改正成本极高」的决策之一。

**读完你会知道:**

- 为什么 django-crontab 和 Celery Beat 两套并存是个持续失血的技术债,以及它是怎么自然形成的
- 如果第一天就选一套,为什么应该选 Celery Beat
- USE_TZ=False 时 Beat 的时区怎么写才不会在半夜错误时间跑任务
- 抓取类定时任务为什么要按「失败是常态」来设计:告警 + 补跑 + 幂等三件套
- 定时任务按频率分层、重任务错峰的实操建议

## 现状:两套系统,几十个任务

我们的定时任务分散在两套调度系统里:

- **django-crontab(CRONJOBS)**:写在 settings 里,本质是把命令注册进系统级 crontab。早期任务大多在这。
- **Celery Beat(CELERY_BEAT_SCHEDULE)**:同样配置在 settings 里,由 Beat 进程按计划把任务投递给 Celery Worker 执行。后来的任务陆续走这套。

两套加起来有几十个任务。没有谁刻意设计成这样——最早项目只有几个 cron 任务,django-crontab 上手最快;后来引入 Celery 做异步任务,新的定时需求顺手挂到 Beat 上;老任务「能跑就不动」,于是就并存至今。

技术债不是某天欠下的,是每天多欠一点。

## 双系统的日常代价

并存的代价不体现在某次大事故,而是摊在每一次日常操作里:

- **排查「任务没跑」永远要查两处**。业务同学说昨天的日报没发,你得先确认这个任务挂在 cron 还是 Beat,再分别去看 crontab 是否注册成功、Beat 进程是否存活、Worker 有没有消费。光是定位「它归谁管」就要花掉前十分钟。
- **改任务要先确认挂在哪一套**。改了 Beat 的调度时间,结果任务在 CRONJOBS 里,改动无效;更糟的是两边各有一个相似任务,改了一边漏了另一边。我们的仓库规范里专门写了一条「改定时任务必须同时检查 CRONJOBS 和 CELERY_BEAT_SCHEDULE」——需要靠纪律弥补的,都是架构欠的。
- **可观测性割裂**。Celery 任务有 Worker 日志、能接监控;cron 任务的输出散落在系统日志里,失败往往无声无息。
- **部署心智负担**。cron 任务改了要重新注册 crontab,Beat 任务改了要重启 Beat 进程——两套生效方式不一样,漏一步就是「改了但没生效」。

## ★教训:第一天就只选一套,选 Celery Beat

这是本页最重要的一句话:**定时任务调度系统,第一天就只选一套,之后所有任务无条件挂这一套。**

我们推荐 Celery Beat,理由很实际:

1. **配置在代码里**,随 git 走。谁在什么时候加了什么任务、改了什么时间,`git log` 一查就有;系统 crontab 的变更历史基本不可考。
2. **有日志、可观测**。任务的投递、执行、异常都走 Celery 的日志与监控体系,失败能看到堆栈,而不是 cron 那种「静默失败,月底才发现」。
3. **和异步任务共用基础设施**。反正你迟早要上 Celery 做异步(发消息、生成报表、调第三方接口都需要),Beat 只是在同一套 Worker/Broker 之上加一个调度进程,不引入任何新组件。
4. **任务就是普通 Celery task**,天然可以手工 `.delay()` 补跑——这一点在后面讲补数据时是救命的。

如果你在用 AI 复刻这套系统:在 M1 骨架阶段(见 [复刻 prompt:M1 骨架](../05-replication/prompts/00-bootstrap.md))就把「定时任务只走 Celery Beat」写进项目的 CLAUDE.md,让 AI 之后每次加定时任务都自动遵守。这一条规则的成本是一行字,收益是永远不用写这一页。

## 时区坑:USE_TZ=False 时,Beat 的小时按本地时间写

Django 的 `USE_TZ = False` 加上时区设为本地时区(我们是 Asia/Shanghai)时,Celery 的时区配置跟着 Django 走,**Beat 里 crontab 的小时字段直接按本地时间写**:想早上 8 点跑,就写 `hour=8`,完事。

我们踩过的坑是「双重换算」:写任务的人凭印象觉得「Celery 内部是 UTC,得减 8 小时」,于是想 8 点跑写成 `hour=0`——但整条链路早已配置成本地时区,结果任务在半夜错误的时间跑了。这类错误特别隐蔽:任务确实跑了、日志也正常,只是时间不对,往往要等业务反馈「数据出来得不对劲」才被发现。

铁律:**时区换算只允许发生在配置层,任务代码和调度表达式里永远写业务意图时间。** 团队里任何人(包括 AI 助手)写 `hour=N` 时,N 就是墙上钟表的时间,不做任何脑内换算。把这条也写进 CLAUDE.md——AI 和新同事一样,会「好心」帮你做 UTC 换算。

时区相关的更多坑(datetime 序列化、跨天边界)见 [后端坑](../03-pitfalls/backend.md)。

## 抓取类任务:失败是常态,不是例外

我们有一类特殊的定时任务:从外卖平台(美团/饿了么/京东)抓取营收等经营数据(详见[外卖平台集成](../02-modules/delivery-platforms.md))。这类任务和普通的「查库—聚合—落库」任务有本质区别:**它依赖你控制不了的外部系统**——对方页面改版、登录态过期、接口限流、网络抖动,任何一个都会让今天的抓取失败。

按「偶尔失败」来设计,你会得到一个每月坑你一次的系统;按「失败是常态」来设计,才能睡得着觉。必须有的两件东西:

- **失败告警**:任务失败或抓到的数据为空/异常时,立刻推送到工作群(我们用飞书机器人),让人当天就知道,而不是月底看报表才发现某几天的数据是空的。判空比判异常更重要——很多失败不抛异常,只是安静地抓回来一份空数据。
- **可补跑设计**:发现某天缺数据后,能用一条命令(或一次手工触发)只补那一天,不影响其他日期的数据。这就引出下一节的幂等。

数据缺口的可怕之处在于发现延迟:缺的是 3 号的数据,发现却在 31 号看月报的时候,这中间平台侧的数据可能已经查不到了。告警把发现延迟从「按月」压缩到「按天」,是这类任务性价比最高的一笔投入。

## 幂等:所有定时任务必须可带日期重跑

我们给所有定时任务立了一条设计规则:**任务函数必须接受日期参数,不传时默认「昨天」或「今天」,传了就处理指定日期;同一天跑两遍,结果必须一样。**

```python
@app.task
def sync_platform_revenue(date_str=None):
    # 不传参 → 默认处理昨天;传 '2026-06-03' → 只补那一天
    target = parse_date(date_str) if date_str else yesterday()
    # 落库用 update_or_create / 先删后插,保证重跑不产生重复数据
    ...
```

这条规则带来的能力,日常几乎全靠它:

- **补数据**:发现 6 月 3 日抓取失败,手工 `sync_platform_revenue.delay('2026-06-03')` 即可(示例日期,非真实数据),不用改代码、不用等下一个调度周期。
- **安全重试**:告警触发后直接重跑,不用先去清理上一次的半成品数据。
- **回溯修复**:口径改了、bug 修了,写个循环把过去 30 天逐日重跑一遍就完成回刷。

反面写法是任务内部隐式取 `now()` 再推算日期、落库直接 `insert`——这种任务失败一次就是永久缺口,重跑一次就是数据翻倍。幂等不是锦上添花,是定时任务的入场券。

## 月度节点型任务:自动跑 + 人工核对的 SOP

有一类任务天然带「账期」属性:每月固定日抓上个月的账单/营收、月度对账、月结报表。这类任务我们不追求全自动,而是写成**人机配合的 SOP**:

1. **自动部分**:每月固定日(比如 1 号)定时任务自动抓取上月数据、生成汇总。
2. **人工部分**:配一份核对清单(checklist),运营同学按清单逐项核对——各平台数据是否齐全、总额和平台后台对得上、异常门店是否已标注。
3. **兜底**:核对发现缺漏,用上一节的日期参数补跑,补完再核对一遍。

为什么不做全自动?因为月度数据直接影响对账和经营决策,错误成本高;而抓取链路(尤其依赖浏览器插件半自动抓取的部分)可靠性有天花板。**在「自动化程度」和「错误成本」之间,月度节点任务应该选可靠而不是选省事。** 一份 10 分钟的人工核对清单,比一套「理论上全自动」但每季度坑你一次的系统便宜得多。

具体到营收数据的完整链路,见[营业额:录入、抓取与达标锁](../02-modules/turnover.md)。

## 任务分层:按频率分类,重任务错峰

任务多了以后(几十个是很快的事),建议按执行频率分层管理,每层有明确的性质约束:

| 层级 | 频率 | 典型任务 | 约束 |
|---|---|---|---|
| 分钟级 | 每 1~10 分钟 | 状态检查、轻量同步 | 只做轻查询,单次执行必须远短于间隔 |
| 小时级 | 每小时/几小时 | 数据聚合、缓存刷新 | 可以做中等计算,注意与分钟级任务错开 |
| 日级 | 每天固定时刻 | 日报生成与推送、对账、抓取 | 放在业务低峰(凌晨/清晨),重任务彼此错峰 |
| 月级 | 每月固定日 | 账期抓取、月结报表 | 走上一节的人机配合 SOP |

两条实操经验:

- **重任务错峰**。日级任务最容易犯的错是全堆在整点(一堆任务都写 `hour=2, minute=0`),数据库和 Worker 瞬时被打满,任务互相拖慢甚至超时。把它们错开到 2:00、2:20、2:40……成本是零,收益是每个任务都跑在资源充裕的窗口里。
- **执行时长要留量**。分钟级任务如果单次执行逼近调度间隔,负载稍高就会出现「上一轮没跑完、下一轮又开始」的堆叠。要么加锁防重入,要么降频,别赌运气。

## 踩坑与红线

- **任务没跑,查了半天调度配置没问题**
  根因:任务挂在另一套调度系统里,你查的这套本来就没有它。
  铁律:调度系统只允许存在一套;历史包袱未清完之前,排查任何定时任务问题必须两套都查。

- **想 8 点跑的任务在半夜跑了**
  根因:链路已配置为本地时区,写任务的人又手工做了一次 UTC 换算,双重换算。
  铁律:USE_TZ=False + 本地时区配置下,crontab 的 `hour` 直接写墙上钟表时间,任何人(含 AI)不做脑内换算。

- **月底看报表才发现月初某几天数据是空的**
  根因:抓取任务静默失败(或抓回空数据不报错),没有告警,发现延迟一个月。
  铁律:抓取类任务必须有失败与空数据告警;数据缺口的发现延迟必须压到「天」级。

- **补跑一次数据,报表数字翻倍**
  根因:任务落库用裸 insert,不幂等,重跑即重复。
  铁律:所有定时任务必须可带日期参数重跑,同一天跑 N 遍结果一致;做不到幂等的任务不允许上线。

- **改了调度时间,任务还按老时间跑**
  根因:两套系统生效方式不同(重注册 crontab vs 重启 Beat 进程),改完漏了生效步骤。
  铁律:改定时任务的 checklist 必须包含「确认生效方式并执行」;这也是只留一套系统的又一个理由。
