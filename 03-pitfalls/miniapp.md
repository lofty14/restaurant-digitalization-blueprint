# 小程序坑:构建 / 发布 / 兼容

> 这一页汇总门店小程序(uni-app + Vue2,编译到微信小程序)踩过的坑。写给要做店端小程序的工程师,以及帮你写代码的 AI 助手——这些坑 AI 也会踩,提前喂给它。

## 读完你会知道

- 小程序和网页开发最大的区别不在语法,在**发布模式**:有人工审核、有客户端缓存、有真机内核差异,每一样都会把你在浏览器里养成的直觉打碎
- 为什么「疑难杂症先全量重编」能省两小时
- 哪些写法在模拟器里一切正常、到安卓真机上直接白屏
- 为什么小程序端代码要写得保守,能放后端的逻辑尽量放后端

## 为什么小程序坑单独成页

管理端前端(网页)出了问题,重新 build、purge 缓存,几分钟内所有用户就是新版本。小程序不一样:

- **发布要过平台人工审核**,通常 1~2 天,紧急 bug 也绕不过去;
- **运行环境是各家手机的 webview 内核**,碎片化程度远超桌面浏览器;
- **构建链路多一层**(uni-app 编译到小程序代码,再由开发者工具/CLI 打包),每一层都可能出鬼;
- **用户是店长店员**,出问题时正在打烊盘点或高峰下单,容错窗口极小。

所以小程序的纪律核心就一句:**把风险前移**——发布前多做,发布后少赌。

下面每条坑按「症状 → 根因 → 铁律」三行式,可以直接抄进你自己项目的检查清单。

## 构建:工具链会骗你

### 增量编译损坏

- **症状**:改了代码,真机/预览上死活没变化;或者报一些与你的改动毫无关系的莫名错误,怎么看代码都没问题。
- **根因**:开发者工具的增量编译产物损坏了——缓存的中间产物和源码对不上,你调试的其实是一份「缝合怪」代码。
- **铁律**:疑难杂症先**全量重编**(清缓存、删编译产物、重新完整构建)再排查。这一步 5 分钟,能省掉对着幻影 bug 排查的两小时。

### 报错凭感觉猜

- **症状**:排查半天,方向全错,最后发现报错信息里早就写明白了。
- **根因**:没先看控制台/真机日志就开始改代码——小程序报错常常发生在编译层或框架层,靠「网页开发的直觉」去猜,大概率猜偏。
- **铁律**:**先看 console 再动手**。模拟器看开发者工具控制台,真机用远程调试/vConsole 看真机日志。报错信息比直觉可靠,这条对人和 AI 同样适用——让 AI 修 bug 时,先把完整报错贴给它,别只描述现象。

## 发布:审核与版本的纪律

### 发的不是最新

- **症状**:明明「发布」了,用户看到的还是旧版本;或者新版本里缺了昨天刚合并的功能。
- **根因**:两种典型翻车——一是把**预览/开发构建**当成发布构建上传了(两者产物不同);二是构建时本地分支不是最新主干,发出去的代码天然缺东西。
- **铁律**:发布必须走正式的**「发行」构建**,不用预览产物;构建前必确认本地已拉最新主干。最好把「拉主干 → 发行构建 → 上传」固化成一条不过脑子的固定流程。

### 发布节奏:审核不可绕过

- **症状**:线上出了紧急 bug,修复代码十分钟写完,但用户要等一两天才能用上——因为要过平台人工审核。
- **根因**:小程序平台的审核机制就是如此,没有「热修复直接生效」这条路,再急也绕不过去。
- **铁律**:**小程序端代码要保守,能放后端的逻辑放后端**(后端随时可发)。具体来说:
  - 判断规则、开关、文案、金额计算,尽量做成后端下发,小程序只负责展示和交互;
  - 小程序端只留「不得不在端上做」的部分,并且写得防御性强一点(字段缺失不崩、接口失败有兜底);
  - 这样多数 bug 只需要发后端,分钟级生效,不用等审核。

## 兼容:模拟器全对,真机全错

### 暗色模式崩样式

- **症状**:部分用户的界面黑白错乱——文字看不清、背景颜色诡异,而你自己手机上一切正常。
- **根因**:用户手机开了系统暗色模式,小程序框架跟着系统切换了组件默认样式,而你的页面根本没为暗色模式设计过。
- **铁律**:业务小程序**直接锁浅色**(配置 `darkmode: false`),别跟系统走。店端工具类小程序没有必要支持暗色,锁死是成本最低、体验最稳的选择。

### 列表 key 崩渲染

- **症状**:安卓真机上列表白屏,或报 WXML 层的渲染错误;模拟器和 iOS 上完全正常。
- **根因**:`v-for` 的 `:key` 用了字符串拼接表达式(比如 `item.a + '-' + item.b`)。编译到 WXML 后,部分安卓内核处理不了这种 key 表达式,直接崩渲染。
- **铁律**:`:key` **只用简单字段**(单个 id 类字段)。发现一处这么写,立刻全局搜同款写法一起改——这种病从来不是只有一处。

### 深分包跳转失败

- **症状**:跳转到分包里的页面时**偶发**白屏,复现不稳定,日志也不明显。
- **根因**:`navigateTo` 对层级较深的分包路径支持不稳定,分包加载与页面栈的时序问题在部分机型上暴露。
- **铁律**:分包间跳转用 `reLaunch` / `switchTab` 兜底,不依赖 `navigateTo` 一条路。宁可牺牲一点页面栈的返回体验,也要保证页面一定打得开。

### 框架 data 静默丢字段

- **症状**:某个 data 字段莫名其妙是 `undefined`,赋值代码明明执行了,视图也不更新。没有任何报错。
- **根因**:旧版框架(Vue2 一系)对**下划线开头**的 data key 不做响应式处理,甚至直接丢弃——这是框架的保留前缀约定,但它不报错,静默吞掉。
- **铁律**:data 字段命名**不用下划线开头**。这类「静默丢弃」的坑最阴,因为没有报错可看,只能靠命名纪律预防。

### 真机纪律

- **症状**:模拟器上全对,发出去真机上全错——样式塌了、白屏、交互没反应。
- **根因**:模拟器用的是桌面内核,真机是各家手机的 webview 内核,渲染和 JS 行为都有差异;上面的 key 崩渲染、深分包白屏,全都是只在真机暴露的。
- **铁律**:**核心流程(下单/盘点/打卡)发布前至少三种真机场景过一遍**——不同品牌安卓 + iOS 是底线。模拟器通过只能算「编译通过」,不算「测试通过」。

## 给 AI 助手的提示

如果你在用 AI 编程助手复刻小程序端,把这页整页喂给它,并额外强调三条:

1. 生成列表代码时,`:key` 一律用单个 id 字段,禁止拼接表达式;
2. 生成 data 结构时,禁止下划线开头的 key;
3. 涉及业务规则的逻辑,先问一句「这段能不能放后端」,默认答案是能。

这三条都是「写的时候多想一秒、上线后少熬一夜」的类型,AI 不知道你的发布要等审核,你得告诉它。
