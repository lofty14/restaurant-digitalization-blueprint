# 踩坑实录:怎么读这一层

> 这一层是我们几年里真实踩过的坑的复盘合集,按端分成四篇;不管你是老板、IT 负责人还是动手写代码的工程师(或者你的 AI 编程助手),开工前都值得先扫一遍。

**读完你会知道:**

- 这一层的四篇分别覆盖什么范围,该按什么顺序读
- 每条坑统一的「症状 → 根因 → 铁律」三行式怎么读、怎么用
- 两种使用姿势:开工前通读混眼熟,出问题时按症状反查
- 为什么四篇里最值钱的是数据口径篇

## 这一层有什么

我们按「坑发生在哪一端」把内容拆成四篇:

| 篇 | 覆盖范围 |
|---|---|
| [后端坑](backend.md) | 时区、数据库迁移、连接管理、JSON 序列化——Django + Celery 技术栈上摔过的跟头 |
| [管理端前端坑](frontend.md) | 缓存、路由、组件——内部管理网页上那些「本地好好的,线上就不对」的问题 |
| [小程序坑](miniapp.md) | 构建、发布、机型兼容——小程序生态特有的、后端工程师最容易低估的一类坑 |
| [数据口径坑](data-caliber.md) | 一个数字在不同页面对不上、一个「在职」有两种定义——最贵的一类坑,单独成篇 |

前三篇是技术坑,第四篇是业务坑。如果你只有时间读一篇,读第四篇。

## 每条坑的统一格式

四篇里的每一条坑,都严格按三行式写:

- **症状**:你看到了什么。用户报障的原话、监控上的表现、页面上的错样子——这是你查问题时的入口。
- **根因**:为什么会这样。把表象往下挖到机制层,讲清楚坑的成因,而不是只给一个「重启就好了」。
- **铁律**:以后怎么办。一句能直接执行的规矩,规定动作,不留判断余地。

举一条真实的坑感受一下这个格式(细节在[后端坑](backend.md)里展开):

> **症状**:新加的定时任务,执行时间总是和预期差 8 小时。
> **根因**:框架默认按 UTC 理解时间,和我们按北京时间写的排程假设不一致。
> **铁律**:项目第一天就选定一套时区策略并写进文档,此后排程时间一律按本地时间写,不做任何心算换算。

铁律之外我们还有个补充习惯:新任务上线后蹲守一次,核对真实执行时间——这不是铁律本体,是给铁律上的保险。

这些铁律都是我们真金白银换来的——每一条背后都是至少一次线上事故或几天的排查。建议直接抄进你团队的开发规范,不用等自己再踩一遍。

## 怎么用这一层

两种姿势,对应两个时机:

1. **开工前通读一遍,混个眼熟。** 不需要背下来。目的是在大脑里埋下钩子:等你哪天遇到「定时任务时间不对」「导出的 Excel 是空的」这类现象时,能想起「好像在哪见过这个坑」。对 AI 编程助手同理——把这四篇喂进上下文,它写代码时就会主动绕开这些雷。
2. **出问题时按症状来查。** 每篇内的条目都按「症状」可检索地命名——你搜的是你看到的现象,不是你猜的原因。排查线上问题时,先来这层用症状关键词搜一圈,命中了就能省掉从零排查的时间。

换句话说:这一层平时是疫苗,出事时是急诊手册。

## 为什么口径篇最值钱

前三篇的技术坑,再糟糕也就是浪费几天:服务重启、缓存清掉、迁移修好,问题就翻篇了。

数据口径的坑不一样。一个字段口径定错——比如订单金额到底读实时价还是下单时的快照价、「在职员工」到底按哪张表哪个字段算——错误会悄悄流进报表、看板、对账单,业务同学基于错的数字做几周甚至几个月的决策,等发现时连「从哪天开始错的」都要考古半天。**技术坑浪费的是天,口径坑浪费的是月。**

所以我们把[数据口径坑](data-caliber.md)单独成篇,并且建议你在动任何统计、报表、财务相关的代码之前,先把它完整读一遍。
