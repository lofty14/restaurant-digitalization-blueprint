# AI 产出的质量纪律

> AI 把产能放大了十倍,也把返工放大了十倍——这页讲我们用真实事故换来的一套质量纪律,适合所有准备让 AI 大规模写代码的团队负责人和工程师读。

**读完你会知道:**

- 为什么大批 AI 并行产出必须由「另一个 AI + 人」对抗复查,而不是生成者自己说了算
- 修一个 bug 之后为什么要立刻全库 grep 同款病,以及不这么做的代价
- 鉴权、迁移、路由这三类改动各自的专属验证动作
- 多个 AI agent 同时在一个仓库干活时,靠什么约定才能不互相踩死
- 「整夜自治」怎么授权、怎么收口,才能睡得着觉

## 先说结论:产能是十倍,纪律不跟上,返工也是十倍

我们的技术团队极小,日常大量代码由 AI 产出——一个晚上并行改几十个文件是常态。这个模式跑了很久之后,我们得到一个朴素的结论:

**AI 的产出速度和它的出错速度是同一个速度。**

人写代码,一天写五百行,错也就错在这五百行里;AI 一夜写五千行,错也可能散在这五千行里。没有配套的质量纪律,你省下的写码时间会加倍还给调试和返工。

下面每一条纪律,都不是从最佳实践文章里抄来的,而是我们真的出过事故之后立的规矩。按「规矩 → 背后的事故」的顺序讲。

## 纪律一:对抗复查——生成者不能自证清白

**规矩:大批 AI 并行产出之后,必须再开一个独立会话的 AI,用「挑刺人设」把改动全部审一遍,人再抽查一部分。生成者与审查者必须是不同的会话。**

为什么强调「不同会话」?因为同一个会话里的 AI 会为自己的产出辩护——它带着生成时的全部上下文和思路,让它自查,它大概率会说「没问题」。换一个干净会话,明确告诉它「你的任务就是挑毛病,找出这批改动里的 bug、遗漏和口径错误」,它挑出来的问题质量完全不同。

我们的实际流程:

1. 一批并行任务(比如十几个 agent 各改一块)跑完;
2. 新开一个审查会话,喂给它改动清单和 diff,人设就是「刻薄的 reviewer」;
3. 审查 AI 输出问题清单,按严重程度排序;
4. 人对高风险改动(鉴权、资金、数据口径)逐个过,低风险的抽查。

这条规矩来自一次真实教训:一次大规模界面优化,几十处并行修复看起来都「完成」了,事后对抗复查才发现其中混着会丢用户状态的改动。从那以后,「大并行产出必对抗复查」成了铁律。

## 纪律二:同款病全库 grep——修一处等于欠九处

**规矩:AI 修好一处 bug 后,立刻用同样的模式全库搜一遍,把所有同款病灶一次修完。**

AI 写代码有个特点:同一个错误写法,它会在很多地方复用。人也一样,但 AI 的复用规模大得多。所以一个 bug 被发现时,几乎可以默认它不是孤例——**同一个病灶散落十处是常态,不是意外。**

我们在小程序上吃过这个亏:一个资源地址处理函数的错误用法,修了报障的那一处以为完事,结果同样的写法在库里还有一批,后续接连爆出同款问题。多轮返工之后立下规矩:修 bug 的最后一步不是提交,而是 grep——把出错的函数名、错误的调用模式当关键词全库搜,搜出来的每一处都判断一遍,能一起修的一起修。

这一步交给 AI 做特别合适:它不嫌枯燥,搜得全,判断得快。你只需要在流程里强制这一步存在。

## 纪律三:鉴权改动双向验证——只测一边等于没测

**规矩:改登录、权限相关的代码,必须同时验证两个方向——「匿名用户被拦住」和「已登录用户仍然通」。**

这条的事故很典型:一次给接口补鉴权,只验证了「没登录的请求确实被拦了」,就放上线了。结果那批接口的调用方根本不在我们以为的用户体系里(门店端有独立的登录态),补上的鉴权把所有正常用户也全拦了——全线「未登录」,业务直接停摆。

根因不是 AI 写错了代码,而是验证只做了一半。鉴权是个双向门:拦得住坏人只是一半,放得过好人才算完整。所以现在规矩是:

- 改鉴权,测试用例必须成对出现:匿名请求 → 预期被拒;合法登录态请求 → 预期正常返回;
- 系统里如果有多套登录态(比如管理端用户和门店端用户走不同体系),每套都要验;
- AI 提交鉴权改动时,必须在说明里贴出两个方向的验证证据。

## 纪律四:迁移与路由要冒烟——最容易互相踩的两类改动

**规矩:数据库迁移先在影子环境空跑;路由/配置类改动改完尽快提交并冒烟验证。**

这两类改动有个共同点:它们是全局共享的单点,多个 agent 并发干活时最容易在这里撞车。

- **迁移**:迁移文件有编号顺序,两个 agent 各自生成迁移就会分叉;迁移内容错了,影响的是全表数据。所以迁移先在影子环境(和生产结构一致的测试库)空跑一遍,确认无冲突、无破坏,再往前走。
- **路由/配置**:路由表是集中式的大文件,agent A 改了没提交,agent B 基于旧版本再改,合并时不是冲突就是覆盖。我们出过并发部署时路由改动互相覆盖的事故,之后的规矩是:**动了路由或全局配置,尽快 commit,并立刻冒烟(把新路由真实请求一遍)**,把「未提交的全局改动」这个窗口压到最短。

## 纪律五:多 agent 共处铁律——并行开发靠约定不靠运气

**规矩:同一仓库多个 agent 并行时,每个 agent 只 `git add` 自己改的文件、绝不 revert 别人的改动、禁止 force push。**

多 agent 并行是我们的常态:一个改前端页面,一个改后端接口,一个刷文档。git 层面它们互相看得见对方的未提交改动,如果没有约定,灾难剧本很好写:agent A 觉得工作区「有脏文件」,顺手 `git checkout .` 清理——agent B 干了一小时的活没了。

所以三条铁律写进了每个 agent 的工作指令:

1. **只 add 自己的文件**——提交时逐个指定文件名,永远不用 `git add .`;
2. **不 revert 别人的改动**——工作区里看不懂的改动,默认是别的 agent 的活,绕开它;
3. **禁 force push**——覆盖远端历史在多人(多 agent)协作里没有任何正当理由。

这些约定听起来像常识,但对 AI 必须显式写下来。AI 默认会「把工作区收拾干净」,这个好习惯在并行场景下就是事故源。

## 纪律六:整夜自治——授权要显式,收口要人工

**规矩:让 AI 通宵干活前,睡前明确划定授权范围;晨起对着改动清单逐项确认,再决定哪些放行。**

我们真的会这么用 AI:睡前把一批优化任务交代下去,AI 通宵跑,早上起来看结果。这个模式很爽,但前提是两头都收紧:

- **睡前(授权)**:明确说清楚哪些事它可以直接改(比如样式微调、文案修正、加日志),哪些事只能出方案等人拍板(比如动数据口径、动鉴权、动资金相关逻辑)。授权必须是显式的白名单,不能是「你看着办」。
- **晨起(收口)**:AI 要产出一份完整的改动清单,人对着清单逐项过——改了什么、为什么改、怎么验证的。没进清单的改动视同违规,直接回滚。

这个模式的本质是:**把「信任 AI」变成一个有边界、可审计的流程**,而不是一句心态。授权范围写得越具体,第二天早上的惊吓越少。

## 纪律七:真验证——「编译通过」不等于「功能对」

**规矩:改完必须在真实环境跑核心流程——真浏览器点一遍、真手机跑一遍;AI 报告「已完成」时,永远要求附证据。**

AI 最容易给人的错觉是「它说做完了,应该就是做完了」。我们反复验证过:**「编译通过」「测试通过」「代码看起来对」三者加起来,仍然不等于「功能对」。** 缓存没清、构建产物是旧的、真机和模拟器行为不一致、旧的前端资源还在用户手里——这些问题任何静态检查都抓不到。

所以验证的标准动作是:

- 前端改动:真浏览器打开页面,把核心流程(登录 → 操作 → 出结果)完整点一遍;
- 小程序改动:真机跑,不能只看开发者工具的模拟器;
- 后端改动:真实请求打到接口,看返回和数据落库;
- **AI 说「已完成」,必须附证据**:截图、请求日志、验证步骤清单,三样至少有一样。没有证据的「已完成」按「未完成」处理。

这条规矩把我们从大量「AI 自信满满、上线一跑就崩」的场景里救了出来。证据要求还有个副作用:AI 为了拿出截图和日志,会被迫真的去跑一遍流程,很多问题在它自己跑的时候就暴露了。

## 踩坑与红线

- **症状**:一夜并行改几十个文件,抽查前几个都没问题,上线后陆续爆雷。
  **根因**:生成者自查形同虚设,同会话 AI 会为自己的产出辩护。
  **铁律**:大并行产出必对抗复查——独立会话的挑刺 AI 全量审 + 人抽查高风险项。

- **症状**:修好报障的那处 bug,一周内同样的问题在别的页面接连复发。
  **根因**:AI(和人)会大规模复用同一个错误写法,病灶天然是散落的。
  **铁律**:修 bug 的最后一步是全库 grep 同款模式,一次修完,再提交。

- **症状**:补完鉴权上线,所有正常用户全线提示「未登录」。
  **根因**:只验证了「匿名被拦」,没验证「已登录仍通」,更没覆盖第二套登录态。
  **铁律**:鉴权改动双向验证,有几套登录体系就验几套,证据随提交附上。

- **症状**:两个 agent 并发干活,一个的路由改动把另一个的覆盖掉,部署后接口 404。
  **根因**:全局单点文件(路由/配置/迁移)在并发下窗口期互相覆盖。
  **铁律**:迁移先影子环境空跑;路由/配置改完尽快 commit 并冒烟。

- **症状**:早上起来发现某个 agent「顺手清理」了工作区,另一个 agent 的半天工作消失。
  **根因**:AI 默认爱收拾工作区,没有多 agent 共处约定。
  **铁律**:只 add 自己的文件、不 revert 别人的改动、禁 force push。

- **症状**:AI 报告「全部完成,测试通过」,用户一用就报错——用户手里跑的还是旧缓存的前端。
  **根因**:把「测试通过」当成了「功能对」,没在真实环境跑核心流程。
  **铁律**:真浏览器/真机验证核心流程;「已完成」必须附截图/日志/清单证据。
