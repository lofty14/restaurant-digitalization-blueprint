# 外卖平台集成:插件半自动抓取模式

> 这页讲我们如何在拿不到外卖平台官方 openAPI 的前提下,用 Chrome 插件半自动抓取商家后台数据,把三大外卖平台的营收、竞对、POS 数据接进自己的系统。适合被「平台不给 API」卡住的中小连锁 IT 负责人和工程师读。

**读完你会知道:**

- 为什么中小连锁基本拿不到外卖平台官方 openAPI,以及现实的替代路径是什么
- Chrome 插件半自动抓取的工作原理,和「一类数据一个插件」的矩阵式拆分
- 平台门店 ID 为什么必须单独建映射表,把 ID 塞进门店主表会付出什么代价
- 月度营收抓取的人机配合 SOP,以及上报接口为什么必须幂等
- 引流品补贴、平台运营费这两类衍生业务怎么挂在抓取数据之上

## 一、现实:官方 openAPI 你大概率拿不到

先泼冷水。美团、饿了么、京东这些外卖平台都有官方开放平台,文档齐全,看起来很美。但真去申请就会发现:开放平台面向的是**服务商**,要企业资质、要服务商认证、要过审核,有的能力还只对头部 ISV 开放。假设一个中小规模的连锁品牌,以自有品牌身份去申请商家数据类接口,大概率走不通,或者拿到的权限远不够用。

我们没有在这条路上耗。判断很简单:**数据在商家后台页面上明明白白摆着,运营同学每天都登录去看**——问题不是「拿不到数据」,而是「怎么把人眼看到的数据自动搬进系统」。

于是有了现实解:**Chrome 插件半自动抓取**。

## 二、方案本体:插件装在运营同学的浏览器里

核心思路一句话:插件装在运营同学日常办公的 Chrome 里,登录态天然存在,插件在商家后台页面上把数据抓下来,调我们自己后端的上报接口入库。

拆开讲几个关键点:

- **登录态不用管**。运营同学本来每天就要登录美团商家版、饿了么商家后台处理业务,插件复用的就是这个浏览器里的会话。不用存密码、不用对抗验证码、不用模拟登录——这是插件方案相对无头爬虫最大的优势,也是「装在真人浏览器里」这个看似土气的决定的全部价值。
- **抓的是自己店的数据**。插件只在商家后台读取本商家有权查看的页面数据,不越权、不碰别人的数据。
- **上报走自家接口**。插件抓到数据后 POST 到我们后端的专用上报接口,后端负责校验、映射、入库。插件本身不做业务逻辑,越薄越好——页面结构会变,插件要改就只改「怎么抓」,不动「抓来干嘛」。
- **半自动,不是全自动**。需要运营同学打开对应后台页面、点一下触发(或按 SOP 在固定日期操作)。我们刻意不追求无人值守——后面「踩坑与红线」会讲为什么。

```mermaid
flowchart LR
    A["运营同学浏览器<br/>已登录平台商家后台"] --> B["Chrome 插件<br/>读取页面数据"]
    B --> C["POST 自家后端<br/>上报接口·幂等"]
    C --> D["平台门店ID映射表<br/>换算成内部门店ID"]
    D --> E["入库<br/>营收/竞对/POS"]
    E --> F["报表·补贴·运营费<br/>等下游业务"]
```

## 三、插件矩阵:一类数据一个插件

我们没有做一个「大而全」的插件,而是**一类数据一个插件**,各自独立安装、独立版本:

| 插件 | 抓什么 | 给谁用 |
|---|---|---|
| 营收账单插件 | 各平台商家后台的月度账单/营收明细 | 负责对账的运营同学 |
| 竞对信息插件 | 平台 C 端可见的竞对门店公开信息 | 做市场分析的同学 |
| POS 数据插件 | 收银系统后台的交易数据 | 门店运营口的同学 |

为什么拆?三个原因:

1. **互不影响**。平台改版只会打挂对应那一个插件,营收抓取挂了不影响竞对采集。
2. **权限最小化**。每个插件只申请自己需要的域名权限,审查和排查都简单。
3. **迭代解耦**。账单页改版频繁,竞对页相对稳定,拆开后各自按需更新,不用整包重发。

代价是插件数量多、分发麻烦一点。对内部工具来说这个代价可以接受——用的人就那几位运营同学,发个新版本群里说一声就装上了。

## 四、平台门店 ID 映射:单独建表,别塞主表(★重点)

这是本页最想让你带走的一条设计决策。

平台侧的门店 ID(比如美团的门店 poi id)和你内部系统的门店 ID 是两套体系,必须做映射。映射有两个天然的复杂性:

- **平台会换 ID**。门店在平台侧重新上单、迁移、换主体,平台 ID 就变了,但内部门店还是那家店。
- **一店多平台**。一家内部门店同时在美团、饿了么、京东各有一个平台 ID,是一对多关系。

我们最早偷懒,把美团门店 ID 直接做成了门店主表上的一个字段。后来痛定思痛,**单独建了映射表**(平台门店 ID ↔ 内部门店 ID,一行一个绑定关系),主表上那个字段标记废弃。

为什么说「痛定思痛」?因为废弃字段的杀伤力比想象中大:**字段还在表里,就永远有人误用**。新同事写查询、AI 助手生成代码,看到主表上有个 `meituan_poi_id` 字样的字段,顺手就用了——而它的数据早就不维护、不准了。我们后来只能在项目文档里反复大写强调「这个字段已废弃,真相源是映射表」,相当于给自己埋了一颗要永远看守的雷。

铁律:**平台侧 ID 映射,从第一天起就单独建表,映射表是唯一真相源**。主表干干净净,谁也误用不了。

## 五、月度营收抓取 SOP:人机配合,幂等兜底

营收账单是抓取业务里最重的一条线,我们把它固化成了月度 SOP:

1. **固定日期执行**。每月固定日期(如每月 1 号,示例安排,按你的对账节奏定),运营同学按操作手册逐平台登录商家后台,用插件抓取上月账单。
2. **逐店上报**。插件把每家门店的月度营收数据打包上报后端,后端按「平台门店 ID → 映射表 → 内部门店 ID」落库。
3. **抓取失败可单店重抓**。网络抖动、页面没加载全、某家店数据异常,都只需要对那一家店重新抓一次,不用全量重来。
4. **上报接口幂等**。这是 SOP 能跑起来的前提:**同一家店同一个月重复上报,是覆盖,不是累加**。后端以「平台 + 内部门店 ID + 月份」为唯一键做 upsert——注意平台维度不能省,一店多平台,少了它,同店同月的美团、饿了么数据会互相覆盖。没有这条,「重抓」就不敢做——谁知道会不会把营收翻倍。

幂等这一条值得单独展开说一句:凡是「人工可能重试」的上报接口,都必须设计成重复调用结果不变。人机配合流程里,人一定会重试——手滑点两次、不确定成没成功再点一次。接口幂等是对操作者最基本的善意,也是数据不炸的保险丝。

## 六、挂在抓取数据之上的两类业务

数据抓回来不是终点,下游至少长出了两类业务。

### 引流品补贴:匹配键用快照

我们对部分引流商品按**商品维度**给门店补贴(平台上卖一单补一点,示例口径)。两个设计点:

- **匹配键要用快照**。补贴规则里锚定的商品名,和平台账单里出现的商品名做匹配。但商品名会改——运营调价改名是常事。所以匹配依据必须是**建规则时的名称快照**,并接受「改名后需人工重新对齐」的运维成本,而不是幻想一个永远自动对上的匹配。
- **平台侧口径定死写进文档**。比如美团侧补贴按**订单量**计(示例口径),这类口径一旦定了就写进文档,所有报表、结算、答疑都引用同一份口径,不允许口头传承。口径的坑我们在[数据口径](../03-pitfalls/data-caliber.md)一页里专门血泪控诉过。

### 平台运营费:规则参数化,不写死

平台评分、费率这些指标是会变的,而它们联动我们内部对门店的收费规则(评分高低对应不同档位的运营费,示例机制)。教训是:**规则参数(评分档位、费率、生效范围)做成可配置,而不是写死在代码里**。平台调整口径、业务调整档位,改配置就生效;写死在代码里,每次调整都是一次发版,而且历史规则难以追溯。

## 踩坑与红线

- **平台页面改版,抓取一夜全挂**
  症状:某天起某平台数据全部缺失,插件点了没反应或抓到空数据。
  根因:插件本质是解析别人家的页面结构,平台前端改版对它就是破坏性变更,这是方案的天生脆弱性,不可消除。
  铁律:必须有失败告警(数据缺失当天就能发现,而不是月底对账才发现)+ 人工兜底流程(手工导出/手工录入通道永远保留);把插件当「随时会坏的东西」运维。

- **追求全自动,越做越脆**
  症状:想去掉人工步骤做成无人值守,结果登录态过期、验证码、风控弹窗轮番出现,维护成本远超收益。
  根因:全自动方案要对抗平台的反自动化机制,而半自动方案借真人的登录态和操作天然绕开了这一切。
  铁律:接受「半自动」的定位——全自动是伪需求,**数据对才是需求**;人点一下按钮的成本,远低于对抗风控的成本。

- **平台 ID 塞进门店主表,废弃后阴魂不散**
  症状:主表上的平台 ID 字段停止维护后,仍不断有新代码(包括 AI 生成的代码)错误引用它,查出来的数据似是而非。
  根因:字段物理存在就构成误用邀请,「文档里写了已废弃」拦不住顺手引用。
  铁律:平台 ID 映射单独建表,从第一天起映射表就是唯一真相源;已铸成的废弃字段,要在项目文档最显眼处反复标注。

- **上报接口累加式入库,重试变灾难**
  症状:某店某月营收莫名翻倍,追查发现运营同学不确定是否上报成功,重新抓了一次。
  根因:上报接口按「插入/累加」实现,没有以「平台 + 店 + 月」为唯一键做覆盖。
  铁律:一切人工可触发的上报接口必须幂等——同键重复上报覆盖而非累加;不幂等,就不敢重试,不敢重试的流程撑不起人机配合。
