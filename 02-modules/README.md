# 业务模块全景与阅读顺序

> 这一层把我们整套系统拆成 14 个业务模块逐一讲透。本页是地图:先看全景表知道每个模块解决什么问题,再按推荐顺序进正文——老板可以只看每页开头的导读,工程师建议顺着主线读完。

## 推荐阅读顺序

```
core-domain → ordering-mall → inventory → points → 其余按兴趣
```

为什么是这个顺序:

- **[core-domain](core-domain.md) 是地基。** 门店、员工、角色权限被后面每一个模块引用,不先搞清"谁在哪家店、有什么权限",读任何模块都会卡壳。
- **[ordering-mall](ordering-mall.md) + [inventory](inventory.md) 是交易主线。** 订货产生单据,单据驱动库存变化,这条链是整套系统里数据量最大、口径要求最严的部分,后面生产、供应链、财务全都挂在这条线上。
- **[points](points.md) 是"唯一出口"的最小样板。** 积分模块业务简单,但把"所有变更走同一个函数"这个模式做得最纯粹——先在这里看懂它,再去读库存和财务里更复杂的版本,会轻松很多。
- **其余按兴趣。** 做运营管理的看 turnover / daily-ops / inspection;做增长的看 site-selection / crm / delivery-platforms;做供应链和财务的看 production-costing / finance-ledger / supply-chain。彼此依赖不强,可以跳读。

## 一个贯穿全层的观察:四个词会反复出现

读的时候留意,几乎每个模块都在重复同一套设计:

1. **状态机**——单据、工单、线索、整改任务,凡是"有生命周期的东西"都用显式状态字段管理流转,禁止用时间戳或布尔值隐式推断状态。
2. **快照**——单据生成时把价格、名称等易变字段冻结成 `snap_*` 快照字段,历史数据绝不读实时值。订货商城是最典型的一页。
3. **唯一出口**——积分、库存这类"账本型"数据,所有增减收敛到一个函数入口,谁绕过谁背锅。这是我们防数据烂账的头号手段。
4. **口径文档**——每个模块配一份说清"数字怎么算"的文档,业务问口径先看文档再看代码。口径没写下来,等于没有口径(为什么这么说,见 [数据口径:最贵的一类坑](../03-pitfalls/data-caliber.md))。

这四个套路不是事先规划出来的,是被坑教出来的——正文里每次出现,基本都能找到一次对应的事故。

## 关于生产与财务两页的特别说明

[production-costing](production-costing.md) 和 [finance-ledger](finance-ledger.md) 涉及我们的商业机密。这两页**只讲结构与模式**:模型怎么拆、状态怎么流转、事件怎么驱动凭证——不含任何真实配方、原料用量、工艺参数,也不含任何真实财务数据。你能学到的是"怎么建这套系统",而不是"我们的配方和账本长什么样"。
